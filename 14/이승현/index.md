# 14장 유튜브 설계

유튜브 시스템은 단순히 비디오를 올리고 보는 것을 넘어, 거대한 트래픽과 데이터를 처리하기 위한 복잡한 기술적 이면을 가지고 있다. 이번 장에서는 대규모 비디오 스트리밍 서비스를 설계해 보겠다.

---

# 1. 문제 이해 및 설계 범위

## 1.1 면접 Q&A로 확정한 요구사항

| 구분 | 내용 |
| --- | --- |
| 핵심 기능 | 비디오 업로드 및 시청 |
| 지원 클라이언트 | 모바일 앱, 웹 브라우저, 스마트 TV |
| 일간 능동 사용자(DAU) | 500만 명 |
| 평균 소비 시간 | 하루 평균 30분 소비 |
| 비디오 제한 | 최대 파일 크기 1GB |
| 주요 품질 | 빠른 업로드, 원활한 재생, 화질 선택 기능, 고가용성 및 안정성 |

## 1.2 개략적 규모 추정
- **신규 저장 용량**: 매일 약 150TB 필요 ($500만 \times 10\% \text{ 사용자} \times 300\text{MB}$).
- **CDN 비용**: 모든 트래픽을 CDN을 통해 서비스할 경우 막대한 과금 발생 ($500만 \times 5\text{회 시청} \times 0.3\text{GB} \times \$0.02 = \$150,000/\text{일}$).
- **최적화**: 비용 절감을 위해 인기 있는 비디오는 CDN에, 나머지는 자체 서버에 두는 등의 전략이 필요함.

# 3. 상세 설계 — 비디오 트랜스코딩

비디오를 다양한 기기에서 재생 가능하게 하고, 대역폭에 따라 화질을 조절하기 위해 트랜스코딩은 필수다.

## 3.1 DAG(유향 비순환 그래프) 모델

창작자의 다양한 요구사항(워터마크, 섬네일 추출, 화질별 인코딩)을 지원하고 병렬 처리를 위해 DAG 모델을 도입한다.

- **전처리기**: 비디오를 GOP(Group of Pictures) 단위로 분할.
- **DAG 스케줄러**: 작업을 단계별로 분할하여 작업 큐에 할당.
- **자원 관리자**: 작업 큐, 서버 큐, 실행 큐를 사용하여 효율적인 자원 배분.
- **작업 서버**: 실제 인코딩 및 이미지 추출 작업 수행.

# 4. 시스템 최적화 방안

## 4.1 속도 및 성능 최적화
- **비디오 병렬 업로드**: 비디오를 GOP 단위로 쪼개어 병렬적으로 업로드하여 속도 향상.
- **업로드 센터 근접 배치**: CDN 에지 서버를 업로드 센터로 활용하여 지연 시간 감소.
- **비동기 처리**: 메시지 큐를 도입하여 모듈 간 결합도를 낮추고 병렬 처리 극대화.

## 4.2 안전성 및 비용 최적화
- **미리 사인된 업로드 URL**: 보안을 위해 허가된 사용자만 특정 위치에 업로드 가능하게 제한(S3 Presigned URL 등).
- **비디오 저작권 보호**: DRM 시스템 도입, AES 암호화, 워터마크 사용.
- **롱테일(Long-tail) 전략**: 인기 비디오는 CDN에서, 인기가 낮은 비디오는 자체 서버에서 처리하여 비용 절감.

---

# 5. 오류 처리 및 확장성

- **회복 가능 오류**: 트랜스코딩 실패 등은 재시도하며, 복구가 어려우면 클라이언트에 오류 코드 반환.
- **장애 대응**: API 서버는 무상태(Stateless)로 설계하여 수평적 확장 보장, DB 및 캐시는 다중화하여 가용성 확보.
- **라이브 스트리밍**: 지연 시간이 매우 중요하므로 낮은 지연 시간을 보장하는 전용 프로토콜과 실시간 인코딩 방식 적용.

---

# 6. 4단계 마무리

이번 설계에서는 유튜브와 같은 비디오 플랫폼의 핵심인 **비디오 전송 저연시간 최소화**와 **거대한 인프라 비용 절감**에 초점을 맞췄다.

- **핵심 요약**: 트랜스코딩의 복잡성을 DAG로 해결하고, CDN과 자체 서버를 혼합하여 비용을 최적화함.
- **추가 논의**: 서비스 규모가 커짐에 따라 데이터베이스 샤딩, API 계층의 추가 확장 등을 고려할 수 있음.
