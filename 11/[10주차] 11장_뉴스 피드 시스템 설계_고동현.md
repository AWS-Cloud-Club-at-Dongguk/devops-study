# [10주차] 11장_뉴스 피드 시스템 설계_고동현

## 1단계: 문제 이해 및 설계 범위 확정

### 주요 요구사항 및 개략적 규모

- **지원 플랫폼:** 모바일 앱 및 웹 브라우저
- **주요 기능:** 새로운 스토리 올리기(피드 발행), 친구들의 스토리 보기(피드 생성)
- **정렬 기준:** 단순 시간 역순(Reverse chronological order)
- **최대 친구 수:** 한 명당 최대 5,000명
- **트래픽 규모:** 매일 천만 명의 능동 사용자(10 million DAU)
- **콘텐츠 유형:** 텍스트뿐만 아니라 이미지, 비디오 등 미디어 파일 포함

## 2단계: 개략적 설계안 제시 및 동의 구하기

뉴스 피드 시스템은 크게 피드 발행(Feed Publishing)과 **뉴스 피드 생성(News Feed Building)** 두 가지 처리 흐름으로 나뉩니다.

### 1. 피드 발행 (Feed Publishing)

사용자가 스토리를 포스팅하면 데이터를 캐시와 데이터베이스에 기록하고, 해당 사용자의 친구들 뉴스 피드에 스토리를 전송합니다.

- **동작 흐름:** 사용자 → 로드밸런서 → 웹 서버 → **포스팅 저장 서비스** / **포스팅 전송(Fanout) 서비스** / **알림 서비스**
- **API:** `POST /v1/me/feed` (바디에 포스팅 내용 포함, 인증 헤더 필요)

### 2. 뉴스 피드 생성 (News Feed Building)

어떤 사용자의 뉴스 피드를 친구들의 포스트를 시간 흐름 역순으로 모아서 만듭니다.

- **동작 흐름:** 사용자 → 로드밸런서 → 웹 서버 → **뉴스 피드 서비스** → 뉴스 피드 캐시
- **API:** `GET /v1/me/feed` (인증 헤더 필요)

## 3단계: 상세 설계

가장 핵심이 되는 포스팅 전송(Fanout) 모델과 캐시 구조에 대한 상세 설계입니다.

### 1. 포스팅 전송 (팬아웃, Fanout) 서비스

어떤 사용자의 새 포스팅을 그 사용자와 친구 관계에 있는 모든 사용자에게 전달하는 과정입니다.

- **쓰기 시점에 팬아웃하는 모델 (Fanout-on-write / Push 모델):**
    - 새로운 포스팅을 기록하는 시점에 뉴스 피드를 미리 갱신합니다.
    - **장점:** 뉴스 피드가 실시간으로 갱신되며, 피드를 읽는 데 드는 시간이 짧아집니다.
    - **단점:** 친구가 많은 사용자의 경우 피드를 갱신하는 데 많은 시간이 소요되는 **핫키(hotkey)** 문제가 발생하며, 자주 로그인하지 않는 사용자의 피드까지 갱신하므로 컴퓨팅 자원이 낭비됩니다.
- **읽기 시점에 팬아웃하는 모델 (Fanout-on-read / Pull 모델):**
    - 피드를 읽어야 하는 시점(On-demand)에 뉴스 피드를 갱신합니다.
    - **장점:** 비활성화된 사용자에게 유리하며, 데이터를 친구에게 푸시하는 작업이 없어 핫키 문제가 발생하지 않습니다.
    - **단점:** 뉴스 피드를 읽는 데 많은 시간이 소요될 수 있습니다.
- **🌟 절충안 (Hybrid 모델):**
    - 대부분의 사용자에게는 빠른 피드 조회를 위해 **푸시(Push) 모델**을 사용하고, 팔로워가 아주 많은 유명인(Celebrity)의 경우 과부하 방지를 위해 팔로워가 피드를 조회할 때 가져가는 **풀(Pull) 모델**을 결합하여 사용합니다.
    - 안정 해시(Consistent Hashing)를 통해 요청과 데이터를 고르게 분산하여 핫키 문제를 완화합니다.

### 2. 뉴스 피드 읽기 흐름과 캐시 아키텍처

이미지나 비디오 같은 미디어 콘텐츠는 CDN에 저장하여 빠르게 읽어갑니다. 시스템의 성능을 좌우하는 것은 캐시이며, 보통 다섯 계층으로 나눕니다.

- **뉴스 피드:** 뉴스 피드의 ID를 보관
- **콘텐츠:** 인기 콘텐츠와 일반 콘텐츠 포스팅 데이터를 보관
- **소셜 그래프:** 사용자 간 관계 정보(팔로워, 팔로잉)를 보관
- **행동(Action):** '좋아요', 답글 등 사용자의 행위 정보 보관
- **횟수(Counter):** '좋아요' 횟수, 답글 횟수, 팔로워 수 등 보관

## 4단계: 마무리

면접 후반부에 규모 확장성과 관련된 추가 논의 주제들입니다.

- **데이터베이스 규모 확장:** 수직적 확장 vs 수평적 확장, SQL vs NoSQL, 주-부(Master-Slave) 다중화, 읽기 연산을 위한 복제본(Replica) 도입, 데이터베이스 샤딩(Sharding).
- **웹 계층의 무상태(Stateless) 운영**.
- **가능한 한 많은 데이터를 캐시할 방법과 여러 데이터 센터 지원 방안**.
- **메시지 큐(Message Queue)를 활용한 컴포넌트 간 결합도 낮추기**.
- **핵심 메트릭 모니터링:** 트래픽이 몰리는 시간대의 QPS, 사용자 피드 새로고침 시의 지연시간 등.
