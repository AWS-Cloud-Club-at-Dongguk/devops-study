# [12주차] 13장_검색어 자동완성 시스템 설계

## 1단계: 문제 이해 및 설계 범위 확정

구글이나 아마존 검색창에 글자를 칠 때마다 찰나의 순간에 추천 검색어가 뜨는 기능은 수천만 명의 사용자가 동시에 타이핑을 하는 상황에서도 빠른 응답 속도를 보장해야 합니다. 설계할 시스템의 요구사항을 다음과 같이 가정합니다.

- 빠른 응답 속도: 사용자가 검색어를 입력할 때, 자동완성 검색어가 100밀리초(ms) 이내에 표시되어야 합니다.
- 연관성: 자동완성되는 검색어는 사용자가 입력한 단어와 연관된 것이어야 합니다.
- 정렬: 시스템의 계산 결과는 인기도 등의 순위 모델(Ranking model)에 의해 정렬되어야 합니다.
- 규모 확장성: 트래픽 규모가 큰 시스템을 감당할 수 있도록 확장 가능해야 합니다. DAU(일간 능동 사용자) 1천만 명 수준을 처리할 수 있어야 합니다.
- 고가용성: 시스템의 일부에 장애가 발생하거나 느려지거나 네트워크에 문제가 생겨도 시스템은 계속 사용 가능해야 합니다.

## 2단계: 개략적 설계안 제시 및 동의 구하기

시스템은 크게 두 부분으로 나뉩니다. 하나는 사용자가 입력한 질의를 실시간으로 수집하는 데이터 수집 서비스이고, 다른 하나는 주어진 접두어(prefix)에 대해 인기 검색어 5개를 반환하는 질의 서비스입니다.

### 1. 초기 설계의 한계: RDBMS LIKE 쿼리

가장 직관적인 방법은 관계형 데이터베이스(RDBMS)에 질의 빈도 테이블을 만들고 SELECT * FROM frequency_table WHERE query LIKE ‘prefix%’ ORDER BY frequency DESC LIMIT 5 와 같은 쿼리를 날리는 것입니다. 데이터 양이 적을 때는 나쁘지 않지만, 천만 명의 사용자가 글자를 타이핑할 때마다 이 쿼리가 데이터베이스로 향한다면 심각한 DB 병목 현상이 발생합니다. 따라서 문자열 탐색에 최적화된 새로운 자료구조가 필요합니다.

### 2. 트라이(Trie) 자료구조

트라이는 글자(character) 노드들을 트리 형태로 연결하여 접두어를 빠르게 찾을 수 있는 자료구조입니다. 하지만 이마저도 최악의 경우 전체 트리를 다 뒤져야 하는 문제가 생길 수 있으므로, 다음 2가지 최적화 기법을 적용합니다.

- 접두어 최대 길이 제한: 검색어는 무한정 길어지지 않습니다. 길이를 상수로 제한하면 노드를 찾는 시간 복잡도를 O(1)로 획기적으로 줄일 수 있습니다.
- 각 노드에 인기 검색어 캐시: 질의가 올 때마다 하위 노드를 탐색하지 않도록, 각 노드에 상위 5~10개의 인기 검색어를 미리 저장(캐시)해 둡니다. 공간은 더 차지하지만 응답 속도를 위해 공간을 희생하는 전형적인 트레이드오프(Trade-off) 전략입니다.

## 3단계: 상세 설계

가장 중요한 핵심 컴포넌트들의 상세 동작 원리입니다.

### 1. 데이터 수집 서비스

사용자가 새로운 단어를 검색할 때마다 실시간으로 트라이 자료구조를 업데이트하는 것은 절대 안 됩니다. 매일 수천만 건의 질의가 들어오는데 그때마다 갱신하면 시스템은 버티지 못합니다. 게다가 자동완성 검색어는 1분 1초마다 바뀔 필요가 없습니다. 따라서 데이터 흐름을 두 가지로 완전히 분리합니다.

- 로그 취합 서버: 사용자의 검색 원본 데이터를 모아서 1주일 등 특정 주기마다 한 번씩 묶어(Aggregation) 비동기적으로 트라이 DB를 갱신합니다.
- 취합된 데이터: 시간 단위 또는 일 단위로 로그를 취합하여, 특정 기간 동안 각 질의가 몇 번 사용되었는지를 기록합니다.
- 작업 서버: 주기적으로 비동기 작업(Job)을 실행하여 트라이 자료구조를 만들고 트라이 DB에 저장합니다.

### 2. 질의 서비스

클라이언트는 트라이 캐시에서 이미 계산된 결과를 빠르게 가져가기만 합니다. 질의 서비스의 성능을 극대화하기 위해 다음과 같은 최적화를 적용합니다.

- AJAX 요청: 요청을 보내고 받기 위해 페이지를 새로고침할 필요가 없으므로 AJAX(Asynchronous JavaScript And XML) 요청을 사용합니다.
- 브라우저 캐싱: 자동완성 검색어 제안 결과는 짧은 시간 안에 자주 바뀌지 않으므로, 브라우저 캐시에 제안 결과를 넣어두면 후속 요청에서 서버 호출 자체를 줄일 수 있습니다.
- 데이터 샘플링: 대규모 시스템에서는 모든 요청의 결과를 로깅하면 CPU 자원과 저장공간을 많이 소모합니다. N개 요청 중 1개만 로깅하는 데이터 샘플링을 적용하여 자원을 아낍니다.

### 3. 트라이 연산

트라이는 매주 한 번씩 갱신됩니다. 갱신 시에는 기존 트라이를 직접 수정하지 않고 새로운 트라이를 만든 뒤 기존 것을 대체하는 방식을 사용합니다. 이 방식의 핵심 연산은 다음과 같습니다.

- 트라이 생성: 작업 서버가 데이터 분석 서비스의 로그나 데이터베이스로부터 취합된 데이터를 이용하여 트라이를 생성합니다.
- 트라이 갱신: 매주 한 번 새로운 트라이를 만들어 기존 트라이를 대체합니다. 오래된 스냅샷(Snapshot)을 백업으로 보관하여 장애 시 복구에 활용합니다.
- 검색어 삭제: 혐오적이거나 폭력적이거나 성적으로 노골적인 질의어 등은 자동완성 결과에서 제거해야 합니다. 트라이 캐시 앞에 필터 레이어(Filter layer)를 두어 부적절한 질의어를 걸러냅니다.

### 4. 규모 확장성 (Scale-out)

데이터가 너무 많아져서 서버 한 대에 트라이를 다 담을 수 없다면 샤딩(Sharding)을 해야 합니다. 알파벳 첫 글자를 기준으로 샤딩하면 구현은 쉽지만, 현실에서는 특정 글자로 시작하는 검색어가 압도적으로 많기 때문에 특정 서버에만 과부하가 걸리는 핫스팟 문제가 발생합니다.

- 샤드 맵 관리자(Shard map manager): 과거의 데이터 패턴을 분석하여 샤드 간의 데이터 양이 균등해지도록 분배하는 별도의 관리자를 두어 해결합니다. 예를 들어, ‘s’로 시작하는 검색어가 ‘u’, ‘v’, ‘w’, ‘x’, ‘y’, ‘z’를 모두 합친 것만큼 많다면 ‘s’만을 위한 샤드를 할당할 수 있습니다.

## 4단계: 마무리 (추가 논의 포인트)

면접의 마지막에 시간이 남는다면 다음과 같은 확장성 및 안정성 문제를 추가로 논의해 볼 수 있습니다.

- 실시간성 반영: 책에서는 주기적인 배치(Batch) 방식으로 데이터를 갱신한다고 했는데, 갑자기 터진 이슈(예: 특정 연예인 뉴스, 재난 상황 등)에 대한 실시간 급상승 검색어는 스파크 스트리밍이나 카프카 같은 스트림 프로세싱을 도입하여 반영할 수 있습니다.
- 다국어 지원: 영어 외에 한국어의 자음/모음 분리 검색 등을 지원하려면 트라이에 유니코드 데이터를 저장할 수 있도록 매핑 방식을 설계해야 합니다.
- 국가별 인기 검색어: 트라이를 CDN(Content Delivery Network)에 저장하여 국가별로 다른 자동완성 결과를 제공하면 응답 지연시간을 개선할 수 있습니다.
- 트렌딩(Trending) 검색어: 순위 모델을 변경하여 최신 검색어에 더 높은 가중치를 부여하는 방식으로 실시간 인기 검색어를 반영할 수 있습니다.
