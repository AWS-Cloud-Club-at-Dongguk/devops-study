# [8주차] 9장_웹 크롤러 설계_고동현

웹 크롤러(Web Crawler)는 검색 엔진에서 널리 쓰는 기술로, 웹에 새로 올라오거나 갱신된 콘텐츠(웹 페이지, 이미지, 비디오, PDF 등)를 찾아내는 것이 주된 목적입니다.

## 웹 크롤러의 주요 용도

- **검색 엔진 인덱싱 (Search Engine Indexing):** 크롤러의 가장 보편적인 용례로, 검색 엔진을 위한 로컬 인덱스를 만듭니다 (예: Googlebot).
- **웹 아카이빙 (Web Archiving):** 나중에 사용할 목적으로 장기보관하기 위해 웹에서 정보를 모으는 절차입니다.
- **웹 마이닝 (Web Mining):** 인터넷에서 유용한 지식을 도출해 냅니다. (예: 주주 총회 자료나 연차 보고서 다운로드)
- **웹 모니터링 (Web Monitoring):** 인터넷에서 저작권이나 상표권이 침해되는 사례를 모니터링합니다.

---

## 1단계: 문제 이해 및 설계 범위 확정

웹 크롤러의 기본 알고리즘은 URL을 다운로드하고, 파싱하여, 새로운 URL을 추출하는 과정을 반복하는 것입니다. 하지만 대규모 시스템에서는 다음과 같은 속성에 주의해야 합니다.

### 좋은 웹 크롤러가 만족시켜야 할 속성

1. **규모 확장성 (Scalability):** 수십억 개의 페이지를 처리하기 위해 병행성(parallelism)을 활용하여 효과적으로 웹 크롤링을 할 수 있어야 합니다.
2. **안정성 (Robustness):** 잘못 작성된 HTML, 아무 반응이 없는 서버, 장애, 악성 코드 등의 비정상적 입력이나 환경에 잘 대응해야 합니다.
3. **예절 (Politeness):** 수집 대상 웹 사이트에 짧은 시간 동안 너무 많은 요청을 보내서는 안 됩니다.
4. **확장성 (Extensibility):** 새로운 형태의 콘텐츠(예: 이미지 파일 등)를 지원하기 쉬워야 합니다.

### 개략적 규모 추정

- 매달 10억 개의 웹 페이지 다운로드 가정
- **QPS (Query Per Second):** 10억 / 30일 / 24시간 / 3600초 = 대략 400 page/sec
- **최대 QPS (Peak QPS):** 2 × QPS = 800
- **저장 용량:** 10억 페이지 × 500KB = 500TB/월. 5년간 보관 시 500TB × 12개월 × 5년 = 30PB 필요.

## 2단계: 개략적 설계안 제시 및 동의 구하기

크롤러의 핵심 컴포넌트와 작업 흐름은 다음과 같습니다.

### 핵심 컴포넌트

- **시작 URL 집합 (Seed URLs):** 웹 크롤링을 시작하는 출발점입니다.
- **미수집 URL 저장소 (URL Frontier):** 다운로드할 URL을 저장 및 관리하는 컴포넌트로, FIFO 큐(Queue) 형태입니다.
- **HTML 다운로더:** 웹 페이지를 다운로드하는 컴포넌트입니다.
- **도메인 이름 변환기 (DNS Resolver):** URL을 IP 주소로 변환합니다.
- **콘텐츠 파서 (Content Parser):** 다운로드한 웹 페이지의 파싱과 검증을 수행합니다.
- **중복 콘텐츠 판별 (Content Seen?):** 해시(Hash) 값을 비교하여 이미 시스템에 저장된 콘텐츠인지 확인합니다.
- **콘텐츠 저장소:** HTML 문서를 보관하는 시스템입니다 (대부분 디스크 저장, 인기 콘텐츠는 메모리 보관).
- **URL 추출기:** HTML 페이지를 파싱하여 상대 경로를 절대 경로로 변환하고 링크를 골라냅니다.
- **URL 필터:** 특정 콘텐츠 타입이나 확장자, 접속 제외 목록 등에 포함된 URL을 크롤링 대상에서 배제합니다.
- **이미 방문한 URL 판별 (URL Seen?):** 블룸 필터(Bloom Filter)나 해시 테이블을 사용하여 이미 처리한 URL인지 추적합니다.
- **URL 저장소:** 이미 방문한 URL을 보관합니다.

## 3단계: 상세 설계

가장 중요한 컴포넌트와 구현 기술의 상세 내용입니다.

### 1. DFS vs BFS

- 웹은 유향 그래프(directed graph)이며, 크롤러는 보통 **BFS(너비 우선 탐색법)**를 사용합니다.
- 하지만 표준적 BFS는 동일한 서버로 수많은 요청을 보내 '예의 없는' 크롤러가 될 수 있으며, URL 간의 우선순위를 두지 않는다는 문제점이 있습니다.

### 2. 미수집 URL 저장소 (URL Frontier)

BFS의 문제점을 해결하기 위해 미수집 URL 저장소를 활용하여 '예의'를 갖추고 '우선순위'를 구별합니다.

- **예의 (Politeness):** 동일 웹 사이트에는 한 번에 한 페이지만 요청하도록, 호스트명(hostname)과 다운로드 수행 작업 스레드(worker thread)를 매핑하여 지연시간(delay)을 두고 처리합니다. (큐 라우터, 매핑 테이블, FIFO 큐, 큐 선택기 활용)
- **우선순위 (Priority):** 페이지랭크(PageRank), 트래픽 양, 갱신 빈도 등을 척도로 URL의 우선순위를 나누어 중요한 페이지를 먼저 수집합니다. (순위결정장치 활용)
- **신선도 (Freshness):** 웹 페이지의 갱신 이력이나 우선순위를 활용하여 주기적으로 재수집(recrawl)을 수행합니다.
- **지속성 저장장치:** 메모리 버퍼와 디스크를 혼합(Hybrid approach)하여 수억 개의 URL을 안정적으로 관리합니다.

### 3. HTML 다운로더와 성능 최적화

- **Robots.txt:** 웹사이트가 크롤러와 소통하는 표준적 방법으로, 수집해도 되는 페이지 목록을 확인합니다. 캐시에 보관하여 성능을 높입니다.
- **성능 최적화 기법:**
    1. **분산 크롤링:** 크롤링 작업을 여러 서버에 분산합니다.
    2. **도메인 이름 변환 결과 캐시:** DNS 요청의 병목을 막기 위해 결과를 캐시하고 크론 잡(cron job)으로 주기적으로 갱신합니다.
    3. **지역성 (Locality):** 크롤링 서버를 크롤링 대상 서버와 지리적으로 가까운 곳에 배치합니다.
    4. **짧은 타임아웃:** 응답이 느린 서버를 무한정 기다리지 않도록 타임아웃을 설정합니다.

### 4. 안정성 및 확장성 확보 전략

- **안정성:** 다운로더 서버 부하 분산을 위한 **안정 해시**(Consistent hashing), 크롤링 상태의 지속적 저장, 예외 처리(Exception handling), 데이터 검증을 적용합니다.
- **확장성:** 새로운 형태의 콘텐츠(예: PNG 파일)를 쉽게 지원할 수 있도록 플러그인 모듈 형태의 아키텍처를 구성합니다.

### 5. 문제 있는 콘텐츠 감지 및 회피

- **중복 콘텐츠:** 해시나 체크섬(checksum)을 사용하여 30%가량의 중복을 걸러냅니다.
- **거미 덫 (Spider Trap):** 크롤러를 무한 루프에 빠뜨리는 구조를 피하기 위해 URL 길이 제한 및 수작업 필터링을 도입합니다.
- **데이터 노이즈:** 광고, 스크립트 코드, 스팸 URL 등 가치 없는 콘텐츠를 제외합니다.

## 4단계: 추가 논의 포인트

- **서버 측 렌더링 (Server-side rendering):** 자바스크립트나 AJAX로 동적 생성되는 링크를 찾기 위해, 파싱 전 동적 렌더링(Dynamic rendering)을 적용합니다.
- **원치 않는 페이지 필터링:** 스팸 방지(anti-spam) 컴포넌트를 두어 저장 공간 및 자원 낭비를 막습니다.
- **데이터베이스 다중화 및 샤딩:** 데이터 계층의 가용성과 규모 확장성을 향상시킵니다.
- **수평적 규모 확장성:** 다운로더 서버를 무상태(stateless) 서버로 만들어 수백, 수천 대로 확장할 수 있게 합니다.
- **데이터 분석 솔루션 (Analytics):** 시스템을 세밀히 조정하기 위해 필수적인 데이터를 수집하고 분석합니다.
