# 09. 웹 크롤러 설계

- 웹 크롤러란
  - 로봇, 스파이더라고도 부름
  - 검색 엔진에서 쓰이는 기술
  - 웹에 새로 올라오거나 갱신된 콘텐츠를 찾아내는 것이 목적
    - 콘텐츠: 웹 페이지, 이미지, 비디오, PDF 파일
  - 몇 개 웹 페이지에서 시작하여 링크를 따라나가며 새로운 콘텐츠를 수집함
    - ![image_1](images/image_1.jpg)
- 용례
  - 검색 엔진 인덱싱
    - 웹 페이지를 모아 검색 엔진을 위한 로컬 인덱스를 만드는 것
    - 예: Googlebot - 구글 검색 엔진이 사용하는 웹 크롤러이다
  - 웹 아카이빙
    - 나중에 사용할 목적으로 장기보관하기 위해 웹에서 정보를 모으는 것
    - 예: 국립 도서관 (미국 국회 도서관, EU 웹 아카이브)
  - 웹 마이닝
    - 웹 마이닝을 통해 인터넷에서 유용한 지식을 도출해낼 수 있음
    - 예: 금융 기업들이 크롤러를 사용하여 주주총회나 연차 보고서를 다운받아 기업의 핵심 사업 방향을 알아내기도 함
  - 웹 모니터링
    - 인터넷에서 저작권이나 상표권이 침해되는 사례를 모니터링
    - 예: 디지마크(Digimarc) - 웹 크롤러를 사용해 해적판 저작물을 찾아내서 보고함
      - * 해적판 저작물: 저작권자의 허락 없이 무단으로 복제, 배포, 판매되는 모든 저작물


## 1. 문제 이해 및 설계 범위 확정
- 웹 크롤러 설계 시, 감당해야 하는 데이터 규모나 기능들을 알아내야 함
- 웹 크롤러의 기본 알고리즘
  - 1. URL 집합이 입력으로 주어지면, 해당 URL들이 가리키는 모든 웹페이지를 다운로드함
  - 2. 다운받은 웹 페이지에서 URL을 추출
  - 3. 추출된 URL들을 다운로드할 URL 목록에 추가하고 1부터 반복
- 이밖에도 규모 확장성을 갖는 웹 크롤러를 면접 시간 내에 설계하기 위해서는 요구사항을 알아내고 설계 범위를 좁히는 과정이 필요함

- 면접관과의 대화를 통해 확정된 요구사항
  - 검색 엔진 인덱스 생성용
  - 매달 10억개의 웹 페이지를 수집해야 함
  - 새로 만들어지거나 수정된 웹 페이지도 고려해야 함
  - 수집한 웹 페이지는 5년간 저장해야 함
  - 중복된 콘텐츠를 갖는 페이지는 무시
- 좋은 웹 크롤러가 만족해야 할 속성
  - 규모 확장성: 웹은 거대하기 때문에, 병행성을 활용하면 효과적임
  - 안정성: 비정상적인 입력이나 환경에 대응 가능해야 함
    - 예: 잘못 작성된 HTML, 아무 반응 없는 서버, 장애, 악성코드가 붙어있는 링크 등
  - 예절: 수집 대상 웹 사이트에 짧은 시간 동안 너무 많은 요청을 보내면 안됨
  - 확장성: 새로운 형태의 콘텐츠를 지원하기 쉬워야 함
    - 예: 이미지 파일을 추가적으로 크롤링하기 위해 전체 시스템을 새로 설계해야 하는 구조는 피해야 함
### 개략적 규모 추정
- 매달 10억개의 웹 페이지를 다운로드
- QPS = 10억/30일/24시간/3600초 = 대략 400페이지/초
- 최대(Peak) QPS = 2 x QPS = 800
- 웹 페이지의 크기 평균은 500k라고 가정
- 10억 페이지 x 500k = 500TB/월
  - 저장 용량: 2장 참고
- 1개월치 데이터를 보관하는 데는 500TB, 5년간 보관한다면 500TB x 12개월 x 5년 = 30PB의 저장용량이 필요할 것
## 2. 개략적 설계안 제시 및 동의 구하기
![image_2](images/image_2.jpg)
### 각 컴포넌트의 기능
- 시작 URL 집합
  - 웹 크롤러가 크롤링을 시작하는 출발점
  - 예: 대학 웹사이트로부터 찾아나갈 수 있는 모든 웹 페이지를 크롤링하는 직관적인 방법은, 해당 대학의 도메인 이름이 붙은 모든 페이지의 URL을 시작 URL로 쓰는 것
  - 전체 웹을 크롤링하는 경우 시작 URL 선정
    - 크롤러가 가능한 한 많은 링크를 탐색할 수 있도록 하는 URL을 골라야 함
    - 일반적으로는 전체 URL 공간을 작은 부분집합으로 나누는 전략을 사용함
      - 지역적인 특색: 나라별로 인기있는 웹 사이트가 다름
      - 주제별로 다른 시작 URL을 사용
      - URL 공간을 쇼핑, 스포츠, 건강 등의 주제별로 세분화하고 각각에 다른 시작 URL을 사용
    - 시작 URL 선정에 정답은 없으므로 해당 시작 URL을 사용한 의도가 무엇인지 명확히 전달하면 됨
- 미수집 URL 저장소
  - 대부분의 현대적 웹 크롤러는 크롤링 상태를 1. 다운로드할 URL, 2. 다운로드된 URL의 두 가지로 나누어 관리
  - 다운로드할 URL을 저장하는 컴포넌트를 미수집 URL 저장소 (URL frontier)라고 부름. FIFO Queue
- HTML 다운로더
  - 인터넷에서 웹 페이지를 다운로드하는 컴포넌트
  - 다운로드할 페이지의 URL은 미수집 URL 저장소가 제공
- 도메인 이름 변환기
  - 웹 페이지를 다운받으려면 URL을 IP 주소로 변환하는 절차가 필요함
  - HTML 다운로더는 도메인 이름 변환기를 사용하여 URL에 대응되는 IP 주소를 알아냄
- 콘텐츠 파서
  - 웹 페이지를 다운로드하면 파싱과 검증 절차를 거쳐야 함
    - 이상한 웹 페이지가 문제를 일으키거나 저장 공간을 낭비하지 않도록..
  - 크롤링 서버 안에 콘텐츠 파서를 구현하면 크롤링 과정이 느려지게 될 수 있으므로, 독립된 컴포넌트로 설계함
- 중복 콘텐츠인가?
  - 약 29%의 웹 페이지 콘텐츠는 중복 -> 같은 콘텐츠를 여러 번 저장하게 될 수 있음
  - 이를 해결하기 위한 자료구조를 도입하여 데이터 중복을 줄이고 데이터 처리에 소요되는 시간을 줄임
    - 이미 시스템에 저장된 콘텐츠임을 알아내기 쉽게 하기 위함
  - 두 HTML 문서를 비교하는 방법
    - 두 문서를 문자열로 보고 비교 
      - 비교 대상 문서의 수가 많은 경우, 느리고 비효율적
    - 웹 페이지의 해시값 비교
- 콘텐츠 저장소
  - HTML 문서를 보관하는 시스템
  - 저장소를 구현하는 데 쓰일 기술 선정: 저장할 데이터의 유형, 크기, 저장소 접근 빈도, 데이터의 유효 기간 등을 종합적으로 고려 필요
  - 이 설계안에서는 디스크와 메모리를 동시에 사용하는 저장소 채택
    - 데이터 양이 너무 많으므로 대부분의 콘텐츠는 디스크에 저장
    - 인기 있는 콘텐츠는 메모리에 두어 접근 지연시간을 줄임
- URL 추출기
  - HTML 페이지를 파싱하여 링크들을 골라내는 역할
  - 링크 추출 사례
    - 상대경로를 전부 https://en.wikipedia.org를 붙여 절대 경로로 변환함
    - ![image_3](images/image_3.jpg)
- URL 필터
  - 다음과 같은 URL을 크롤링 대상에서 배제
    - 특정 콘텐츠 타입이나 파일 확장자를 갖는 URL
    - 접속 시 오류가 발생하는 URL
    - 접근 제외 목록(deny list)에 포함된 URL
- 이미 방문한 URL?
  - 이를 구현하기 위해, 이미 방문한 URL이나 미수집 URL 저장소에 보관된 URL을 추적할 수 있도록 하는 자료구조 이용
  - 이미 방문한 적 있는 URL인지 추적하면, 같은 URL을 여러 번 처리하는 일을 방지할 수 있음
    - -> 서버 부하를 줄이고 시스템이 무한 루프에 빠지는 일을 방지할 수 있음
  - 자료 구조: 블룸 필터, 해시 테이블 이용
- URL 저장소
  - 이미 방문한 URL을 보관하는 저장소
### 웹 크롤러 작업 흐름
![image_4](images/image_4.jpg)
1. 시작 URL들을 미수집 URL 저장소에 저장
2. HTML 다운로더는 미수집 URL 저장소에서 URL 목록을 가져옴
3. HTML 다운로더는 도메인 이름 변환기를 사용하여 URL의 IP 주소를 알아내고, 해당 IP 주소로 접속하여 웹 페이지를 다운받음
4. 콘텐츠 파서는 다운된 HTML 페이지를 파싱하여 올바른 형식을 갖춘 페이지인지 검증
5. 콘텐츠 파싱과 검증이 끝나면 중복 콘텐츠인지 확인하는 절차를 거침
6. 중복 콘텐츠를 확인하기 위해서, 해당 페이지가 이미 저장소에 있는지 확인
   - 이미 저장소에 있는 콘텐츠인 경우, 처리하지 않고 버림
   - 저장소에 없는 콘텐츠인 경우, 저장소에 저장한 뒤 URL 추출기로 전달
7. URL 추출기는 해당 HTML 페이지에서 링크를 골라냄
8. 골라낸 링크를 URL 필터로 전달
9. 필터링이 끝나고 남은 URL만 중복 URL 판별 단계로 전달
10. 이미 처리한 URL인지 확인하기 위해 URL 저장소에 보관된 URL인지 확인함. 이미 저장소에 있는 URL은 버림
11. 저장소에 없는 URL은 URL 저장소에 저장할 뿐 아니라 미수집 URL 저장소에도 전달

## 3. 상세 설계
- DFS vs BFS
  - 웹은, 페이지가 노드이고 하이퍼링크(URL)이 edge인 directed graph
  - 크롤링 프로세스는 이 directed graph를 edge를 따라 탐색하는 과정
  - DFS: 웹처럼 그래프 크기가 클 경우 어느 정도로 깊숙이 가게 될 지 가능이 어려움
  - BFS: FIFO 큐를 사용하는 알고리즘으로, 웹 크롤러에 보통 사용됨
    - 큐의 한쪽으로는 탐색할 URL을 집어넣고 다른 한쪽으로는 꺼내기만 함
    - 문제점
      - 한 페이지에서 나오는 링크의 상당수는 같은 서버로 되돌아감
        - ![image_5](images/image_5.jpg)
        - 위와 같이, wikipedia.com에서 추출한 모든 링크는 모두 내부 링크(동일한 wikipedia.com 서버의 다른 페이지를 참조하는 링크)
        - 이때 크롤러는 같은 호스트에 속한 많은 링크를 다운받게 되고, 이를 병렬로 처리하게 된다면 위키피디아 서버에 수많은 요청이 걸림 -> 과부하
        - -> '예의 없는(impolite)' 크롤러
      - 표준적 BFS 알고리즘은 URL 간에 우선순위를 두지 않음
        - 처리 순서에 있어 모든 페이지를 공평하게 대우
        - 하지만 모든 웹 페이지가 같은 수준의 품질, 중요성을 갖지 않음
        - -> 페이지 순위 (page rank), 사용자 트래픽의 양, 업데이트 빈도 등 여러 척도에 비추에 우선순위를 구별해야 함
- 미수집 URL 저장소
  - 미수집 URL 저장소를 활용하면 위 문제들을 해결할 수 있음
  - 이 저장소를 잘 구현하면 예의를 갖춘 크롤러, URL 사이의 우선순위와 신선도(freshness)를 구별하는 크롤러를 구현할 수 있음
  - 예의
    - 웹 크롤러는 수집 대상 서버로 짧은 시간 안에 너무 많은 요청을 보내는 것을 삼가야 함
    - 너무 많은 요청을 보내는 것은 무례한(impolite) 일이며, 때로는 DoS(Denial-of-Service) 공격으로 간주되기도 함
    - 아무런 안전장치가 없는 웹 크롤러는 초당 수천 건의 페이지 요청을 동일한 웹 사이트로 보내어 사이트를 마비시킬 수도 있음
    - 예의바른 크롤러를 만드는 데 있어서 지켜야 할 원칙
      - 동일한 웹 사이트에 대해서는 한 번에 한 페이지만 요청
      - 같은 웹 사이트의 페이지를 다운받는 태스크는 시간차를 두고 실행하도록 하면 됨
      - 이를 위해서, 웹사이트의 호스트명과 다운로드를 수행하는 작업 스레드 사이의 관계를 유지하면 됨
      - 즉, 각 다운로드 스레드는 별도 FIFO 큐를 가지고 있어서, 해당 큐에서 꺼낸 URL만 다운로드 함
      - ![image_6](images/image_6.jpg)
        - 큐 라우터: 같은 호스트에 속한 URL은 언제나 같은 큐로 가도록 보장하는 역할
        - 매핑 테이블: 호스트 이름과 큐 사이의 관계를 보관하는 테이블
        - FIFO 큐: 같은 호스트에 속한 URL은 언제나 같은 큐에 보관됨
        - 큐 선택기: 큐들을 순회하면서 큐에서 URL을 꺼내서, 해당 큐에서 나온 URL을 다운로드하도록 지정된 작업 스레드에 전달
        - 작업 스레드: 전달된 URL을 다운로드. 전달된 URL은 순차적으로 처리될 것이며, 작업들 사이에는 일정한 지연시간(delay)을 둘 수 있음
  - 우선순위
    - 예: 애플 제품에 대한 사용자 의견이 올라오는 페이지가 애플 홈페이지와 같은 중요도를 갖지는 않음
    - 크롤러 입장에서는 중요한 페이지 (애플 홈페이지)를 먼저 수집하도록 하는 것이 바람직함
    - 유용성에 따라 URL의 우선순위를 나눌 때는 페이지랭크(PageRank), 트래픽 양, 갱신 빈도 등의 척도 이용
    - ![image_7](images/image_7.jpg)
      - 순위 결정 장치: URL 우선순위를 정하는 컴포넌트. URL을 입력으로 받아 우선순위를 계산
      - 큐: 우선순위별로 큐가 하나씩 할당됨. 우선순위가 높으면 선택될 확률도 오라감
      - 큐 선택기: 임의 큐에서 처리할 URL을 꺼내는 역할 담당. 순위가 높은 큐에서 더 자주 꺼내도록 프로그램되어 있음
  - ![image_8](images/image_8.jpg)
    - 위 두 내용을 반영한 전체 설계
    - 전면 큐: 우선순위 결정 과정을 처리
    - 후면 큐: 크롤러가 예의 바르게 동작하도록 보증
  - 신선도
    - 웹 페이지는 수시로 추가, 삭제, 변경됨 -> 데이터의 신선함(freshness)을 유지하기 위해서는 이미 다운로드한 페이지라고 해도 주기적으로 재수집(recrawl)할 필요가 있음
    - URL 재수집 최적화 전략
      - 웹 페이지의 변경 이력 이용
      - 우선순위를 활용하여, 중요한 페이지는 좀 더 자주 재수집
    - 미수집 URL 저장소를 위한 지속성 저장 장치
      - 검색 엔진을 위한 크롤러의 경우, 처리해야 하는 URL의 수가 아주 많음.
        - 이 모두를 메모리에 보관하면: 안정성, 규모 확장성에 좋지 않음
        - 전부 디스크에 저장: 느려서 성능 병목지점이 되기 쉬움
      - 절충안: 대부분의 URL은 디스크에 두지만 IO 비용을 줄이기 위해 메모리 버퍼에 큐를 둠
        - 버퍼에 있는 데이터는 주기적으로 디스크에 기록 
- HTML 다운로더
  - HTTP 프로토콜을 통해 웹 페이지를 내려받음
  - 로봇 제외 프로토콜 (Robots.txt)
    - 웹사이트가 크롤러와 소통하는 표준적 방법
    - 이 파일에 크롤러가 수집해도 되는 페이지 목록이 포함됨
    - 웹 사이트를 긁어 가기 전에 크롤러는 해당 파일에 나열된 규칙을 먼저 확인해야 함
    - Robots.txt를 주기적으로 다시 다운받아 캐시에 보관 
    - 예: https://www.amazon.com/robots/txt 
      - ![image_9](images/image_9.jpg)
      - CreatorHhub와 같은 디렉터리의 내용은 다운받을 수 없다는 내용
  - 성능 최적화
    - HTML 다운로더에 사용할 수 있는 성능 최적화 기법들
    - 1. 분산 크롤링
      - 성능을 높이기 위해 크롤링 작업을 여러 서버에 분산
      - 각 서버는 여러 스레드를 돌려 다운로드 작업 처리
      - URL 공간을 작은 단위로 분할하여, 각 서버는 그중 일부의 다운로드를 담당하도록 함
      - ![image_10](images/image_10.jpg)
    - 2. 도메인 이름 변환 결과 캐시
      - 도메인 이름 변환기 (DNS Resolver)는 크롤러 성능 병목 중 하나
      - DNS 요청을 보내고 결과를 받는 작업의 동기적 특성 때문
      - DNS 요청의 결과를 받기 전까지는 다음 작업을 진행 불가
      - DNS 요청이 처리되는 데는 10ms~200ms 소요
      - 크롤러 스레드 중 어느 하나라도 이 작업을 하고 있으면 다른 스레드의 DNS 요청은 전부 블록됨
      - -> DNS 조회 결과로 얻어진 도메인 이름과 IP 주소 사이의 관계를 캐시에 보관해놓고, 크론 잡(cron job) 등을 돌려 주기적으로 갱신하도록 해 놓으면 성능 향상 가능
    - 3. 지역성
      - 크롤링 작업을 수행하는 서버를 지역별로 분산하는 방법
      - 크롤링 서버가 크롤링 대상 서버와 지역적으로 가까우면 페이지 다운로드 시간 감소
      - 지역성(locality)을 활용하는 전략은 크롤 서버, 캐시, 큐, 저장소 등 대부분의 컴포넌트에 적용 가능
    - 4. 짧은 타임아웃
      - 응답이 느리거나 아예 응답하지 않는 웹 서버의 경우, 최대 얼마나 기다릴지를 미리 정해두는 것
      - 이 시간 동안 서버가 응답하지 않으면 크롤러는 해당 페이지 다운로드를 중단하고 다음 페이지로 넘어감
- 안정성 확보 전략
  - 시스템 안정성을 향상시키기 위한 접근법
    - 안정 해시: 
      - 다운로더 서버들에 부하를 분산할 때 적용 가능한 기술
      - 다운로더 서버를 쉽게 추가하고 삭제 가능
      - 5장 '안정 해시 설계'
    - 크롤링 상태 및 수집 데이터 저장:
      - 장애가 발생한 경우에도 쉽게 복구할 수 있도록 크롤링 상태와 수집된 데이터를 지속적 저장장치에 기록
      - 저장된 데이터를 로딩하고 나면 중단되었던 크롤링을 쉽게 재시작 가능
    - 예외 처리:
      - 예외가 발생해도 전체 시스템이 중단되지 않고 작업이 이어지도록 할 것
    - 데이터 검증:
      - 시스템 오류를 방지하기 위한 중요 수단
- 확장성 확보 전략
  - 새로운 형태의 콘텐츠를 쉽게 지원할 수 있도록 설계 해야 함
    - ![image_11](images/image_11.jpg)
      - 새로운 모듈을 끼워넣어 새로운 형태의 콘텐츠를 지원할 수 있도록 설계
      - PNG 다운로더: PNG 파일을 다운로드하는 플러그인 모듈
      - 웹 모니터: 웹을 모니터링하여 저작권이나 상표권이 침해되는 일을 막음
- 문제 있는 콘텐츠 감지 및 회피 전략
  - 중복, 무의미, 유해한 콘텐츠 감지 및 시스템으로부터 차단
    - 1. 중복 콘텐츠
      - 해시, 체크섬을 사용하여 중복 콘텐츠 탐지
    - 2. 거미 덫
      - 크롤러를 무한 루프에 빠뜨리도록 설계한 웹 페이지
      - 예: spidertrapexample.com/foo/bar/foo/bar/foo/...
        - 무한히 깊은 디렉터리 구조를 포함하는 링크
        - URL의 최대 길이를 제한하면 회피할 수 있음
      - 가능한 모든 종류의 덫을 피할 수 있는 만능 해결책은 없음
      - 덫이 설치된 웹 사이트인지 알아내는 것은 쉽지만 (이상하리만치 많은 웹 페이지를 가지고 있음)
      - 덫을 자동으로 피해가는 알고리즘을 만드는 것은 까다로움
      - 사람이 수작업으로 덫을 확인하고 찾아낸 후, 덫이 있는 사이트를 크롤러 탐색 대상에서 제외하거나 URL 필터 목록에 걸어두면 피할 수 있음
    - 3. 데이터 노이즈 
      - 가치 없는 콘테츠: 광고, 스크립트 코드, 스팸 URL 등
      - 크롤러에게 도움되지 않으므로 가능하다면 제외
## 4. 추가 논의 사항
- 서버 사이드 렌더링
  - 많은 웹사이트가 자바스크립트, AJAX 등의 기술을 사용해서 링크를 즉석에서 만들어냄
  - 웹 페이지를 있는 그대로 다운받아 파싱해보면 그렇게 동적으로 생성되는 링크는 발견할 수 없음
  - 페이지를 파싱하기 전에 서버 사이드 렌더링(= 동적 렌더링)을 하면 해결할 수 있음
- 원치 않는 페이지 필터링
  - 크롤링에 소요되는 자원(저장 공간 등)은 유한하기 때문에, 스팸 방지(anti-spam) 컴포넌트를 두어 품질이 조악하건나 스팸성인 페이지를 걸러내도록 하면 좋음
- 데이터베이스 다중화 및 샤딩
  - 다중화, 샤딩 등의 기법을 적용하면 데이터 계층의 가용성, 규모 확장성, 안정성이 향상됨
- 수평적 규모 확장성
  - 대규모의 크롤링을 위해서는 다운로드를 실행할 서버가 아주 많이 필요할 수 있음
  - 수평적 규모 확장성을 달성하는 데 중요한 것은, 서버가 상태정보를 유지하지 않도록 하는 것 (stateless 서버로 맏느는 것)
- 가용성, 일관성, 안정성
  - 성공적인 대형 시스템을 만들기 위해 고려해야 함 (1장)
- 데이터 분석 솔루션
  - 데이터 수집 및 분석이 중요함
  - 시스템을 세밀히 조정하기 위해서는 데이터와 분석 결과가 필수적임
