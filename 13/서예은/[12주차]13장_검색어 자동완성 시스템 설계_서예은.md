# 13장 검색어 자동완성 시스템

구글 검색이나 아마존 웹사이트 검색창에 단어를 입력하다 보면, 입력 중인 글자에 맞는 검색어가 자동으로 완성되어 표시되는 것을 볼 수 있다. 이런 기능은 보통 **검색어 자동완성**(autocomplete, typeahead, search-as-you-type, incremental search)이라 부른다. 예를 들어 구글 검색창에 `dinner`를 입력하면 `dinner ideas`, `dinner recipes`, `dinner near me` 등이 제안으로 뜬다. 검색어 자동완성은 많은 제품에서 중요한 기능이다. 이번 장에서는 **가장 많이 이용된 검색어 k개**를 자동완성하여 출력하는 시스템을 설계한다.

---

# 1. 문제 이해 및 설계 범위 확정

## 1.1 설계 범위

본 설계에서 전제로 두는 조건은 다음과 같다.

- 자동완성은 검색어 문자열의 **접두어(prefix, 첫 부분)**에 대해서만 수행한다. **중간 부분** 일치는 다루지 않는다.
- 한 번에 표시할 자동완성 검색어 개수는 **5개**이다.
- 5개를 고르는 기준은 **질의 빈도**에 따른 검색어 **인기 순위**이다.
- **맞춤법 검사·자동수정**은 제공하지 않는다.
- 질의 언어는 **영어**를 기본으로 하며, 시간이 허락하면 **다국어 지원**을 확장 논의할 수 있다.
- 모든 질의는 **영어 소문자**만 사용한다고 가정한다(대문자·특수문자 별도 처리 없음).
- **일간 능동 사용자(DAU)** 규모는 **천만(10 million) 명**이다.

---

## 1.2 요구사항 정리

시스템이 충족해야 할 요구사항은 다음과 같다.

- **빠른 응답 속도:** 사용자가 검색어를 입력함에 따라 자동완성 검색어도 충분히 빨리 표시되어야 한다. 페이스북 검색어 자동완성 시스템에 관한 문서를 보면 시스템 응답 속도는 **100밀리초 이내**여야 한다. 그렇지 않으면 시스템 이용이 불편해진다.
- **연관성 (Relevance):** 자동완성 검색어는 사용자가 입력한 단어와 **관련**이 있어야 한다.
- **정렬 (Sorting):** 시스템 계산 결과는 **인기도** 등 **랭킹 모델**에 따라 정렬되어야 한다.
- **규모 확장성 (Scalability):** 높은 트래픽을 감당할 수 있도록 확장 가능해야 한다.
- **고가용성 (High Availability):** 일부가 실패하거나 느려지거나 예기치 않은 네트워크 문제가 있어도 시스템이 동작해야 한다.

---

## 1.3 개략적 규모 추정

- DAU: **천만 명** 가정.
- 사용자 1명당 하루 평균 검색 횟수: **10회** 가정.
- 질의당 평균 입력 데이터량: **20바이트** 가정.
  - ASCII 인코딩(문자 1개 = 1바이트).
  - 질의는 평균 **4단어**, 단어당 평균 **5글자** → \(4 \times 5 = 20\) 바이트.

검색창에 **글자를 입력할 때마다** 클라이언트가 자동완성 백엔드로 요청을 보낸다고 하면, 검색 **한 번**당 평균 **20번**의 요청이 백엔드로 간다. 예: `dinner`를 입력하면 다음 순서로 요청이 간다.

- `search?q=d`
- `search?q=di`
- `search?q=din`
- `search?q=dinn`
- `search?q=dinne`
- `search?q=dinner`

**평균 QPS**는 대략 **24,000**이다.

\[
\frac{10{,}000{,}000 \times 10 \times 20}{24 \times 3600} \approx 24{,}000
\]

- **피크 QPS** ≈ 평균 QPS × 2 ≈ **48,000** 가정.
- 질의 중 약 **20%**가 새로운 검색어라고 가정하면, 하루 **약 0.4GB**의 신규 데이터가 시스템에 추가된다.

\[
10{,}000{,}000 \times 10 \times 20 \times 20\%
\]

# 2. 개략적 설계안 제시 및 동의 구하기

대략 시스템은 두 부분으로 나눌 수 있다.

1. **데이터 수집 서비스 (Data Gathering Service):** 사용자가 입력한 질의를 **실시간**으로 모으는 시스템. 데이터 양이 매우 큰 애플리케이션에는 실시간 방식이 이상적이지 않을 수 있으나, 설계의 출발점으로는 적절하고 상세 설계에서 더 현실적인 방안으로 바꾼다.
2. **질의 서비스 (Query Service):** 주어진 질의에 대해 **가장 인기 있는 검색어 5개**를 정렬해 반환하는 서비스.

---

## 2.1 데이터 수집 서비스

**빈도 테이블(frequency table)**에 질의와 사용 빈도를 저장한다고 가정한다. 처음에는 비어 있다. 사용자가 순서대로 `twitch`, `twitter`, `twitter`, `twilio`를 검색하면 테이블 상태는 다음처럼 변한다.

1. 초기: 비어 있음 (`질의` / `빈도` 열).
2. `twitch` → `twitch | 1`
3. `twitter` → `twitch | 1`, `twitter | 1`
4. `twitter` → `twitch | 1`, `twitter | 2`
5. `twilio` → `twitch | 1`, `twitter | 2`, `twilio | 1`

---

## 2.2 질의 서비스

아래와 같은 빈도 테이블이 있다고 하자. 필드는 **`query`**, **`frequency`** 두 가지다.

| query          | frequency |
| :------------- | :-------- |
| twitter        | 35        |
| twitch         | 29        |
| twilight       | 25        |
| twin peak      | 21        |
| twitch prime   | 18        |
| twitter search | 14        |
| twilio         | 10        |
| twin peak sf   | 8         |

이 상태에서 사용자가 검색창에 **`tw`**를 입력하면, 위 빈도 테이블 값을 사용해 계산한 **top 5** 자동완성 검색어는 예를 들어 다음과 같다.

- twitter
- twitch
- twilight
- twin peak
- twitch prime

**top 5**는 아래와 같은 SQL로 구할 수 있다.

```sql
SELECT * FROM frequency_table
WHERE query Like 'prefix%'
ORDER BY frequency DESC
LIMIT 5
```

데이터 양이 적을 때는 괜찮은 설계이나, 데이터가 매우 커지면 데이터베이스가 병목이 될 수 있다. 이 문제는 **상세 설계**에서 다룬다.

---

# 3. 상세 설계

개략 설계의 두 부분(데이터 수집 서비스 + 질의 서비스)만으로는 최적화된 결과라고 하기 어렵지만, 좋은 출발점이다. 여기서는 다음 순서로 일부 컴포넌트를 더 상세히 설계하고 최적화 방법을 논한다.

- 트라이(trie) 자료구조
- 데이터 수집 서비스
- 질의 서비스

---

## 3.1 트라이 자료구조

개략적 설계안에서는 관계형 데이터베이스를 저장소로 썼으나, **가장 인기 있는 다섯 개 질의**를 골라내는 방식으로는 효율적이지 않다. 해결책으로 **트라이(trie, 접두어 트리 prefix tree)**를 사용한다. 트라이는 시스템의 핵심 부분이므로 요구사항에 맞는 트라이를 만들 시간을 충분히 할애해야 한다. (참고 문헌 [2], [3]의 아이디어 차용.)

**특징·정의**

- 문자열을 간략하게 저장할 수 있는 자료구조다.
- 이름은 "retrieval"에서 왔으며, 문자열을 꺼내는 연산에 초점을 맞춘 설계다.
- **핵심 아이디어**
  1. 트라이는 **트리** 형태의 자료구조다.
  2. **루트 노드**는 **빈 문자열**을 나타낸다.
  3. 각 노드는 **글자(character) 하나**를 저장하며, **26개**(해당 글자 다음에 등장할 수 있는 모든 글자 수)의 자식 노드를 가질 수 있다.
  4. 각 트리 노드는 하나의 **단어** 또는 **접두어 문자열(prefix string)**을 나타낸다.

**저장 예시:** 질의어 `tree`, `try`, `true`, `toy`, `wish`, `win`이 보관된 트라이의 개념적 구조.

- **root**
  - **t**
    - **tr**
      - **tre** → 완성 단어 **tree**
      - **tru** → 완성 단어 **true**
      - **try** → 완성 단어 **try**
    - **to** → 완성 단어 **toy**
  - **w**
    - **wi**
      - **wis** → 완성 단어 **wish**
      - **win** → 완성 단어 **win**

**인기 순 정렬을 위해** 노드에 **빈도 정보**를 저장해야 한다. 예시 빈도 테이블:

| query | frequency |
| :---- | :-------- |
| tree  | 10        |
| try   | 29        |
| true  | 35        |
| toy   | 14        |
| wish  | 25        |
| win   | 50        |

이 빈도 정보를 트라이 노드에 저장하면, 각 노드에 인기 검색어와 점수가 반영된 형태가 된다.

---

## 3.2 트라이로 자동완성 구현하기

알고리즘을 보기 전 용어를 정의한다.

- \(p\): 접두어(prefix)의 길이
- \(n\): 트라이 전체 노드 개수
- \(c\): 특정 노드의 **자식 노드** 개수

**가장 많이 사용된 질의 k개를 찾는 알고리즘**

1. **접두어 노드 찾기:** 주어진 접두어를 나타내는 노드를 찾는다. 시간 복잡도 \(O(p)\).
2. **서브트리 순회:** 접두어 노드부터 서브트리를 순회해 **유효 노드**를 모은다. 유효 노드는 완전한 검색 문자열을 이루는 노드다. 시간 복잡도 \(O(c)\).
3. **정렬 후 top k:** 유효 노드를 정렬해 인기 상위 k개를 고른다. 시간 복잡도 \(O(c \log c)\).

**예:** \(k = 2\)이고 사용자가 접두어 **`be`**를 입력했다고 하자.

1. 접두어 노드 `be`를 찾는다.
2. `be`부터 서브트리를 순회해 유효 노드를 찾는다. 예: `[beer: 10]`, `[best: 35]`, `[bet: 29]`.
3. 정렬해 상위 2개를 고른다 → **`best`**, **`bet`**.

**전체 시간 복잡도:** \(O(p) + O(c) + O(c \log c)\).

직관적이지만, 최악의 경우 **전체 트라이**를 뒤져야 할 수 있다. 이를 완화하기 위한 최적화:

1. **접두어 최대 길이 제한**
2. **각 노드에 인기 검색어 캐시**

---

## 3.3 최적화: 접두어 최대 길이 제한

사용자가 아주 긴 검색어를 입력하는 경우는 드물다. 접두어 최대 길이를 \(p\) (예: **50자**)로 두어도 된다.

접두어 최대 길이를 제한하면 **접두어 노드를 찾는 단계**의 시간 복잡도가 \(O(p)\)에서 **작은 상수에 해당하는 \(O(1)\)**에 가깝게 된다.

---

## 3.4 최적화: 노드에 인기 검색어 캐시

서브트리 전체를 도는 대신, **각 노드**에 그 노드를 통과하는 **인기 검색어 k개 목록**을 저장한다. 대부분 시스템은 제안을 **5~10개**만 보여 주므로 \(k\)는 작다.

- **장점:** top 5 조회 시간 복잡도가 크게 줄어든다.
- **단점:** 모든 노드에 캐시 목록을 두므로 **저장 공간**이 추가된다. 다만 응답 속도 향상이 보통 그 비용을 상쇄한다.

**캐시가 붙은 트라이 예시 (개념)**

- 루트에서 `b` → `[best: 35, bet: 29, bee: 20, be: 15, buy: 10]` 등
- `be` → `[best: 35, bet: 29, bee: 20, be: 15, beer: 10]` 등
- `bu` → `[buy: 14]`
- `bee` → `[bee: 20, beer: 10]`
- `bes` → `[best: 35]`
- `bet` → `[bet: 29]`
- `beer` → `[beer: 10]`
- `best` → `[best: 35]`
- `buy` → `[buy: 14]`
- `w` → `[win: 11]`, `wi` → `[win: 11]`, `win` → `[win: 11]`

위 두 최적화를 적용하면:

1. 접두어 노드 찾기: **\(O(1)\)**
2. 인기 검색어 top 5: 노드에 이미 캐시되어 있으므로 **\(O(1)\)**

전체적으로 **\(O(1)\)**에 가깝게 된다.

## 3.5 데이터 수집 서비스 (재설계 동기)

이전 설계는 **매 키 입력마다** 트라이를 갱신하는 식에 가깝다. 실제로는 비현실적인 이유가 두 가지 있다.

1. **성능:** 일일 수천만 건 규모에서 매 순간 트라이를 갱신하면 검색 서비스가 심각하게 느려진다.
2. **변경 빈도:** 인기 검색어는 트라이가 한번 잡히면 그다지 자주 바뀌지 않는다. 끊임없이 갱신할 필요가 없다.

확장 가능한 데이터 수집은 데이터가 **어디서 오는지**, **어떻게 쓰이는지**에 따라 달라진다. 트위터 같은 **실시간** 앱은 최신 제안이 중요하고, 일반 검색 엔진은 그만큼 자주 갱신할 필요가 없을 수 있다. 어떤 경우든 공통 기반은 트라이용 데이터가 보통 **데이터 분석 서비스** 또는 **로깅 서비스**에서 온다는 점이다.

---

## 3.6 데이터 파이프라인 개요

흐름(개념):

`데이터 분석 서비스 로그` → `로그 취합 서버` → `취합된 데이터` → `작업 서버` → `트라이 데이터베이스`

- `트라이 데이터베이스`는 **매주** 갱신되고, **매주** 그 상태를 스냅샷하여 `트라이 캐시`를 갱신한다는 식의 관계로 설명된다.

### 데이터 분석 서비스 로그

- 검색창에 입력된 질의에 관한 **원본 데이터**가 보관된다.
- 새 데이터가 **추가**될 뿐 **수정**은 없다.
- 로그 데이터에는 **인덱스를 걸지 않는다.**

예시:

| query | time                |
| :---- | :------------------ |
| tree  | 2019-10-01 22:01:01 |
| try   | 2019-10-01 22:01:05 |
| tree  | 2019-10-01 22:01:30 |
| toy   | 2019-10-01 22:02:22 |
| tree  | 2019-10-02 22:02:42 |
| try   | 2019-10-03 22:03:03 |

### 로그 취합 서버

- 분석 서비스 로그는 양이 매우 크고 형식도 제각각인 경우가 많다. **취합(aggregation)**하여 시스템이 소비하기 쉬운 형태로 만든다.
- 취합 주기는 용례에 따라 다르다. 트위터 같은 실시간 앱은 취합 주기를 짧게 가져갈 수 있다.
- **대부분의 경우** 일주일에 한 번 정도 취합해도 충분하다. 설계 시 **취합 주기를 얼마나 짧게 가져갈지**는 서비스 특성(실시간성 요구)에 따라 달라진다. 본 설계안은 **일주일 주기**로 취합해도 충분하다고 가정한다.

### 취합된 데이터

- `time` 필드: 해당 **주가 시작한 날짜**를 나타낸다.
- `frequency` 필드: 해당 질의가 그 주에 사용된 **횟수의 합**이다.

예시:

| query | time       | frequency |
| :---- | :--------- | :-------- |
| tree  | 2019-10-01 | 12000     |
| tree  | 2019-10-08 | 15000     |
| tree  | 2019-10-15 | 9000      |
| toy   | 2019-10-01 | 8500      |
| toy   | 2019-10-08 | 6256      |
| toy   | 2019-10-15 | 8866      |

### 작업 서버

- **작업 서버(worker)**는 주기적으로 **비동기 작업(job)**을 실행하는 서버 집합이다.
- 트라이 자료구조를 만들고 **트라이 데이터베이스**에 저장하는 역할을 담당한다.

### 트라이 캐시

- **분산 캐시 시스템**으로 트라이 데이터를 **메모리**에 유지해 읽기 성능을 높인다.
- **매주** 트라이 데이터베이스의 스냅샷을 떠서 갱신한다.

### 트라이 데이터베이스

## 3.7 트라이 저장 방식

선택지는 크게 두 가지다.

1. **문서 저장소 (Document Store)**  
   매주 새 트라이를 만들므로 트라이를 주기적으로 **직렬화**해 DB에 넣을 수 있다. **MongoDB** 같은 문서 저장소에 저장하기 쉽다.

2. **키-값 저장소 (Key-Value Store)**  
   트라이를 해시 테이블 형태로 바꾼다.
   - 트라이에 저장된 **모든 접두어**를 해시 테이블 **키**로 변환한다.
   - 각 트라이 노드에 저장된 데이터를 **값**으로 변환한다.

**매핑 예:** 접두어 `b` → `be` → `bee` / `bes` … 구조와 노드별 인기 목록이 있을 때, 노드 하나가 `<키, 값>` 한 쌍이 된다.

| 키   | 값                                    |
| :--- | :------------------------------------ |
| b    | [be: 15, bee: 20, beer: 10, best: 35] |
| be   | [be: 15, bee: 20, beer: 10, best: 35] |
| bee  | [bee: 20, beer: 10]                   |
| bes  | [best: 35]                            |
| beer | [beer: 10]                            |
| best | [best: 35]                            |

키-값 저장소 동작이 불명확하면 **6장 키-값 저장소 설계**를 참고한다.

---

## 3.8 질의 서비스 (개선안)

개략 설계의 질의 서비스는 DB에서 top 5를 골랐다. 아래는 그 비효율을 개선한 흐름이다.

1. **사용자 단말**(웹 브라우저, 모바일 앱)
2. **로드밸런서**
3. **API 서버**
4. **트라이 캐시**에서 조회
5. **캐시 미스** 시 **트라이 데이터베이스**에서 읽어 캐시를 채운다.

**단계 요약**

1. 검색 질의가 로드밸런서로 간다.
2. 로드밸런서가 API 서버로 보낸다.
3. API 서버가 트라이 캐시에서 데이터를 가져와 자동완성 응답을 만든다.
4. 캐시에 없으면 DB에서 가져와 캐시에 넣는다. 다음에 같은 접두어 요청이 오면 캐시로 처리한다. 캐시 미스는 **메모리 부족**이나 **캐시 서버 장애**에서도 발생할 수 있다.

---

## 3.9 질의 서비스 최적화

질의 서비스는 매우 빨라야 하다.

- **AJAX 요청:** 웹에서는 보통 AJAX로 자동완성 목록을 가져온다. 페이지를 새로 고치지 않고 요청·응답할 수 있다.
- **브라우저 캐싱:** 자동완성 제안은 자주 바뀌지 않는 경우가 많아 **브라우저 캐시**에 두면 이후 요청을 캐시에서 처리할 수 있다.
  - 구글 검색 예: `system design interview` 검색 시 캐싱 메커니즘 사용.
  - 응답 헤더 예: `cache-control: private, max-age=3600` → **개인 캐시**, **3600초(1시간)** 유효. `private`은 공유 캐시가 아닌 단일 사용자용 응답임을 의미한다.
- **데이터 샘플링:** 대규모에서는 모든 질의 결과를 로깅하면 CPU·저장소 부담이 크다. 요청 중 일부만(예: N개 중 1개) 로깅하는 **샘플링**을 쓸 수 있다.

---

## 3.10 트라이 연산

트라이는 자동완성 시스템의 핵심 컴포넌트다.

- **트라이 생성:** 작업 서버가 분석 서비스 로그나 DB에서 취합된 데이터로 수행한다.
- **트라이 갱신**
  1. **주기적 전체 갱신:** 예를 들어 **일주일에 한 번** 완전히 새 트라이를 만들어 기존 것을 교체한다.
  2. **개별 노드 갱신:** 노드 하나씩 갱신. **성능이 좋지 않아** 본 설계에서는 선택하지 않았으나 작은 트라이에는 맞을 수 있다. 한 노드를 갱신하면 **조상 노드**에 저장된 인기 검색어 결과도 함께 갱신해야 한다. 예: `beer` 빈도가 10→30이면 `beer` 노드뿐 아니라 조상 노드들의 기록도 맞춰야 한다.

---

## 3.11 검색어 삭제

혐오·폭력·성적 노골 등 **위험한 질의**는 자동완성 결과에서 제거해야 한다. 방법으로 **트라이 캐시 앞에 필터 계층(filter layer)**을 두어 부적절한 질의가 반환되지 않게 한다. 필터 규칙에 따라 검색 결과를 유연하게 바꿀 수 있다는 장점이 있다. DB에서 해당 검색어를 **물리적으로 삭제**하는 것은 **다음 갱신 주기**에 **비동기**로 진행하면 된다.

**흐름 (개념):** API 서버 → 필터 계층 → 트라이 캐시.

---

## 3.12 저장소 규모 확장 (샤딩)

트라이가 한 서버에 들어가기 너무 클 때를 대비한다.

### 첫 글자 기준 샤딩

영어만 지원하면 **첫 글자**로 샤딩할 수 있다.

- 서버 **2대**: `a`~`m`으로 시작하는 검색어는 1번 서버, 나머지는 2번 서버.
- 서버 **3대**: `a`~`i` → 1번, `j`~`r` → 2번, 나머지 → 3번.

영어 알파벳이 26자이므로 이 방식만 쓰면 서버는 **최대 26대**로 제한된다.

### 계층적 샤딩

더 늘리려면 샤딩을 **계층적으로** 한다. 첫 글자는 1단계, 두 번째 글자는 2단계 샤딩에 쓴다. 예: `a`로 시작하는 검색어를 **4대**에 나눌 때 — `aa`~`ag` → 1번, `ah`~`an` → 2번, `ao`~`au` → 3번, 나머지 → 4번.

### 데이터 편중

`'c'`로 시작하는 단어가 `'x'`로 시작하는 것보다 훨씬 많다는 점을 고려하면, 서버에 **균등 분배**가 어렵다.

### 검색어 대응 샤드 관리자 (shard map manager)

과거 질의 데이터 **패턴을 분석**해 샤딩하는 방안을 제안한다. **검색어 대응 샤드 관리자**는 어떤 검색어가 **어느 저장소 서버**에 있는지 정보를 관리한다. 예: `'s'`로 시작하는 검색어 양이 `'u','v','w','x','y','z'`를 합친 것과 비슷하다면, `'s'`용 샤드 하나와 `'u'`~`'z'`용 샤드 하나를 두는 식으로 조정할 수 있다.

**흐름 (개념):** 웹 서버가 샤드 관리자에서 **접두어에 해당하는 샤드 위치**를 조회한 뒤, 해당 **샤드**에서 데이터를 읽는다.

# 4. 마무리

## 4.1 확장·운영 시 추가로 고려할 점

- **다국어:** 비영어 문자는 **유니코드**로 트라이에 저장한다. 유니코드는 전 세계 문자 체계를 포괄하는 표준 인코딩이다.
- **국가별 인기 순위:** 지역마다 인기 검색어가 다르면 **국가별로 트라이를 분리**해 두고, **CDN**에 배치하면 응답 시간을 줄일 수 있다.
- **실시간 트렌드 반영:** 주기적(예: 주 1회) 작업 서버와 트라이 재구축 위주의 설계는 **즉각적인** 트렌드 반영에는 맞지 않을 수 있다. 갱신 주기·트라이 빌드 시간이 병목이 된다.

## 4.2 실시간에 가깝게 가져가려면

- **샤딩**으로 한 번에 처리하는 데이터량을 줄인다.
- **랭킹 모델**을 조정해 **최근** 검색어에 더 큰 가중을 둔다.
- 데이터를 **스트림**으로 보고 **스트림 처리** 인프라를 검토한다: Apache Hadoop MapReduce, Apache Spark Streaming, Apache Storm, Apache Kafka 등.
