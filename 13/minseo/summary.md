### 1. 전체 구조 한 방 정리

이 시스템은 크게 3단계로 나뉜다:

1) 로그 수집 (Raw Data)
- 사용자가 입력한 검색어를 그대로 저장
- 수정 없음 / 추가만 있음 (append-only)
2) 데이터 가공 (Batch 처리)
- 로그 → 주기적으로 모아서
- 검색어별 빈도 계산
- 그걸 기반으로 트라이(Trie) 생성
3) 서비스 (Query 처리)
- 사용자가 "tw" 입력하면
- 트라이에서 top 5 검색어 즉시 반환

** 요약 구조 **
사용자 → 로그 → 집계 → 트라이 생성 → 캐시 → API → 자동완성

### 2. 핵심 설계 포인트 (시험/면접 핵심)
#### 2-1. 왜 DB가 아니라 Trie 쓰냐?
- DB에서 LIKE 'tw%' → 느림
- 정렬 + 필터 → 비용 큼

- Trie 특징
  - prefix 기반 검색 → 매우 빠름
  - O(1)에 가까운 응답 가능 (캐싱 시)

#### 2-2. Trie 최적화 핵심 2개
1. prefix 길이 제한
- 최대 길이 제한 -? 탐색 O(1)
2. 각 노드에 top k 캐싱
- 이미 결과 저장 -> 검색 시 계산 필요 없음

** 결과 **
자동완성 응답 = 거의 O(1)

#### 2-3. 왜 실시간 업데이트 안 함?

- 검색어 인기 = 자주 안 바뀜
- 매 입력마다 업데이트 → 성능 폭발

- -> 그래서:

- 로그만 쌓고
- 주기적으로 배치 처리
1. 시스템 흐름 (실제 동작)
2) 사용자 입력
- d → di → din → dinner: 매 입력마다 요청 발생

2) API 서버
- Trie 캐시 조회
3) 캐시 있으면
- 바로 응답

4) 캐시 없으면

- DB 조회 → 캐시에 저장

### 4. 로그 시스템 설계 핵심

- 로그 특징
- append-only
- 수정 없음
- 엄청 큰 데이터
- 분석용

-> 그래서: 로그 = 빠르게 쓰는 게 핵심
