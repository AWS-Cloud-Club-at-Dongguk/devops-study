# 로그 데이터에 인덱스를 안 거는 이유
  1. 쓰기 성능 최우선
    > 인덱스 있으면:
    insert 할 때마다 인덱스 업데이트 발생
    → 쓰기 속도 급감

    > 로그는 초당 수만 건 쓰기 발생
    → 인덱스 = 병목

  2. 로그는 조회 대상이 아님
    > 실시간 조회 안 함. 그냥 쌓아두고 나중에 분석
    > 즉: 로그는 "저장용"이지 "검색용"이 아님
  3. 배치 처리에서 다 읽음
    > 어차피 전체 scan
인덱스 의미 없음
한 줄 핵심
    > 로그 = 쓰기 최적화 구조

    > 인덱스 = 읽기 최적화 구조

→ 방향이 완전히 반대라서 안 씀

# 아파치 카프카 (Apache Kafka)
- 대용량 로그/이벤트를 실시간으로 처리하는 분산 스트리밍 플랫폼
- 역할

    - 사용자 이벤트 → Kafka → 로그 처리 시스템
    - 핵심 개념
      1) Producer
      데이터를 보내는 쪽
      예: 사용자 검색 이벤트
      2) Broker
      Kafka 서버
      데이터를 저장
      3) Topic
      데이터 카테고리
      ex) search-log
      4) Consumer
      데이터를 읽는 쪽
      예: 로그 분석 시스템
    - 특징 4가지
      ① 초고속 처리
      초당 수백만 이벤트 가능
      ② append-only 구조
      로그랑 동일 구조
      ③ 확장성
      파티션 기반 분산 처리
      ④ 장애 내성 replication 있음
- 왜 Kafka 쓰냐?

    - 로그 시스템에서:
데이터 폭발적으로 들어옴
실시간 처리 필요

    - Kafka는:
버퍼 역할
스트리밍 처리 가능


#### Kafka = 로그 데이터의 고속 파이프라인

# 캐싱 없이도 Trie가 빠른 이유
# 1. 결론 먼저

👉 캐싱 없이도 Trie가 빠른 이유:

```text
DB → "데이터 전체에서 조건을 찾는다" (검색 기반)
Trie → "경로를 따라간다" (구조 기반)
```

즉,

* DB = 비교 / 필터링
* Trie = 직접 접근

---

# 2. DB 방식 (왜 느려지는가)

자동완성 쿼리를 DB로 하면 보통 이렇게 된다:

```sql
SELECT * 
FROM queries 
WHERE query LIKE 'tw%' 
ORDER BY frequency DESC 
LIMIT 5;
```

## 내부에서 일어나는 일

### 1) prefix 조건 검사

* 모든 데이터 또는 인덱스 범위 탐색
* 문자열 비교 수행

👉 비용: O(N)

---

### 2) 정렬 (ORDER BY)

* 조건 맞는 데이터 정렬

👉 비용: O(N log N)

---

### 3) 상위 5개 추출

* 그중 top k 선택

---

## 핵심 문제

👉 “tw” 찾으려고 전체 데이터 스캔 가능

---

# 3. Trie 방식 (왜 빠른가)

Trie는 완전히 다르게 동작한다.

## 예시

검색어: "tw"

---

## 1) 문자 따라 내려감

```text
root → t → w
```

👉 이건 비교가 아니라 “경로 이동”

👉 비용: O(p) (p = prefix 길이)

---

## 2) 해당 노드에서 결과 가져옴

* 이미 그 아래 데이터가 구조적으로 묶여 있음

---

## 핵심 차이

```text
DB: "tw로 시작하는거 어디있지?" → 찾음
Trie: "tw 위치로 바로 이동" → 끝
```

---

# 4. 핵심 비교 (진짜 중요)

| 구분        | DB     | Trie        |
| --------- | ------ | ----------- |
| 탐색 방식     | 조건 검색  | 경로 탐색       |
| 시간 복잡도    | O(N)   | O(p)        |
| prefix 처리 | 문자열 비교 | 구조적으로 이미 분리 |
| 데이터 구조    | flat   | 계층형         |

---

# 5. 직관적인 비유

## DB

👉 전화번호부에서 "김" 찾기

* 전체를 보면서
* "김"인지 확인

---

## Trie

👉 "ㄱ → ㅣ → ㅁ" 경로로 바로 이동

* 이미 정렬된 구조
* 찾는 게 아니라 “도달”

---

# 6. 왜 prefix 검색에서 특히 강력한가

Trie는 **prefix 기준으로 데이터를 쪼개서 저장**한다.

예:

```text
t
 ├── w
 │    ├── twitter
 │    ├── twitch
 │    └── twillo
```

👉 "tw"면:

* 이 subtree만 보면 끝

👉 DB는:

* 전체에서 필터링

---

# 7. 캐싱 없어도 빠른 이유 정리

### ① 탐색 범위 자체가 다름

* DB: 전체 or 큰 범위
* Trie: 특정 경로

---

### ② 문자열 비교 없음

* DB: LIKE → 문자열 비교
* Trie: 문자 하나씩 이동

---

### ③ 데이터가 이미 그룹화됨

* Trie: prefix 기준으로 묶여 있음
* DB: 묶여 있지 않음

---

# 8. 그래도 Trie의 단점 (균형 있게)

### ① 메모리 많이 씀

* 노드 많음

### ② 업데이트 어려움

* 변경 시 구조 영향 큼

### ③ 복잡한 쿼리 불가능

* 범위 검색, 조건 검색 약함
---
# Trie가 대체 뭐야?
```markdown
Trie는 "개념"이 아니라

👉 메모리에 올라가는 실제 자료구조(객체 구조)다
```

즉,

* 추상 개념 ❌
* DB 같은 저장소 ❌
* 파일 형식 ❌

👉 **코드로 만든 트리 구조 (객체들의 집합)**

---

# 📌 실제로는 어떻게 저장되냐?

## 1. 메모리 구조 (진짜 핵심)

Trie는 보통 이렇게 구현된다:

```python
class TrieNode:
    def __init__(self):
        self.children = {}   # 다음 문자들
        self.is_end = False  # 단어 끝 여부
        self.top_k = []      # 인기 검색어 캐시
```

👉 실제 메모리 상태:

```text
[Node]
 ├── children → hash map (char → Node)
 ├── is_end → bool
 └── top_k → list
```

---

## 2. 실제 저장 형태

👉 메모리 안에서는:

```markdown
노드 객체들이 서로 포인터로 연결된 트리 구조
```

즉:

```text
root (주소)
 ├── 't' → Node 주소
      ├── 'w' → Node 주소
           ├── ...
```

👉 이건 완전히 **포인터 기반 그래프 구조**

---

# 📌 중요한 포인트

## Trie는 "값"이 아니라 "구조"다

DB:

```text
"twitter", 100
```

Trie:

```text
t → w → i → t → t → e → r (경로 자체가 데이터)
```

---

좋은 질문이다.

## 1. 평소에는 메모리에 있음

👉 빠른 조회를 위해

---

## 2. 디스크에는 이렇게 저장됨

### 방법 1: 직렬화 (serialize)

```json
{
  "t": {
    "w": {
      "top_k": ["twitter", "twitch"]
    }
  }
}
```

👉 MongoDB / 파일 / key-value store

```text

```

# 📌 왜 메모리에 두냐?

👉 이유:

```markdown
디스크 접근 = 느림
메모리 접근 = 빠름

→ 자동완성은 ms 단위 요구
→ 무조건 메모리
```
