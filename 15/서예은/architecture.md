# 내부·파일·객체 저장

---

## 후보 한줄 소개

| 이름                | 이런 것                                                                                                          |
| ------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **MinIO**           | S3 API 비슷하게 쓰는 **오픈소스 오브젝트 스토리지**. 단일 서버로도 쓰고, 노드를 늘려 **분산(EC 등)** 으로도 씀   |
| **NAS**             | 장비랑 OS가 같이 나오는 **완제품 파일 서버**. SMB/NFS로 드라이브처럼 쓰는 그림                                   |
| **Linux 파일 서버** | 일반 서버에 NFS, Samba(SMB), FTP, **Nextcloud** 같은 거 올려서 **파일 공유**만 하는 구성                         |
| **Ceph**            | 여러 대를 묶어 **S3 수준 오브젝트 스토리지**를 직접 깔 수 있는 **분산 스토리지**. 규모·난이도 둘 다 큼           |
| **ZFS + 파일 서버** | 한 대에 디스크 여러 개를 **ZFS 풀**로 묶고, 그 위를 NFS/SMB로 내보내는 **고급 단일 파일 서버**                   |
| **GlusterFS**       | 서버 여러 대 디스크를 묶어 **큰 파일시스템 하나처럼** 쓰는 **분산 파일시스템** (오브젝트 스토어와는 느낌이 다름) |

## MinIO — 서버를 몇 대 두고 어떻게 굴릴지 (개념만)

고민: **한 대냐 여러 대냐**, **복제·패리티(EC)** 를 어떻게 가져가냐, **백업을 따로 두냐**.

**후보 1 — 2대: 운영 1 + 백업 1**

```
[Server 1] MinIO 운영
[Server 2] 백업/복제
```

- 장점: 단순하고 **가볍게** 시작하기 좋음. **수십 TB** 급에는 현실적인 편.
- 단점: 운영 서버 한 대가 죽으면 **서비스가 멈출 수 있음**(HA라고 부르긴 애매). **수백 TB**로 키우면 빨리 빡빡해질 수 있음.

**후보 2 — 2대: 둘 다 저장 + 서로 복제**

```
[Server 1] 저장소
[Server 2] 저장소 + 복제본
```

- 장점: 데이터가 한쪽만 있지 않아서 **조금 더 안심**.
- 단점: 같은 데이터를 두 번 쓰니까 **용량**이 들고, “MinIO **분산 클러스터**”라기보단 **미러/복제**에 가깝다.

**후보 3 — 3대: 3노드 분산**

```
[Node 1]  [Node 2]  [Node 3]
```

- 장점: 2대보다 **견고해지고**, 용량도 나눠 담기 쉬움. **수백 TB** 보려면 현실적인 시작선으로 많이 언급됨.
- 단점: 네트워크·디스크 설계 부담. 3대는 **최소 구성**이라 아주 여유 있다고 보긴 어렵다.

**후보 4 — 3대: 운영 2 + 백업/아카이브 1**

```
[Server 1] 운영
[Server 2] 운영
[Server 3] 백업/아카이브
```

- 장점: **서비스용**이랑 **오래 보관용**을 나눌 수 있음. 수십 TB에는 무난한 편.
- 단점: **수백 TB**만 운영 두 대로는 빡빡할 수 있음. “완전한 분산 객체만” 보기엔 절충형.

**요청이 흐르는 그림 (아주 단순)**

```
클라이언트 (S3 SDK 등)
  → HTTP (S3 API)
  → MinIO
  → 디스크 (분산이면 여러 노드)
```

- **좋은 점**: AWS S3랑 비슷한 API라 **연동 자료가 많음**. 분산이면 **EC(이레이저 코딩)** 같은 걸로 나눠 담고 복구하는 그림도 가능.
- **귀찮은 점**: 노드·디스크 늘릴 때 **클러스터 규칙**을 맞춰야 함.

## NAS

- **뭐냐면**: 네트워크에 붙는 **완제품 파일 서버**(하드웨어 + 제조사 OS).
- **좋은 점**: 꽂고 켜면 비슷하게 쓰기 시작할 수 있고, **GUI**, 스냅샷·백업 기능이 패키지로 있는 경우가 많음.
- **아쉬운 점**: 디스크·슬롯 꽉 차면 **장비 단위**로 갈아타야 하는 경우가 많음. 앱 입장에선 **경로·마운트** 위주라, **웹 서비스가 S3 SDK만** 쓰기로 했을 때랑은 사용 모델이 다름.

**이런 계열이 있다**

- 하드웨어 중심: **Synology**, **QNAP**
- 소프트 중심(서버에 OS 올림): **TrueNAS**(ZFS 기반) 등

---

## 일반 Linux 파일 서버

- NFS, **Samba(SMB)**, FTP/SFTP, **Nextcloud / Owncloud** 같은 걸로 “공유 폴더”를 만든다.
- 보통은 **한 덩어리 서버** 스토리. 2대 쓴다고 해서 자동으로 HA가 되는 건 아니고, **동기화·장애 조치**는 따로 생각해야 함.

| 소프트웨어           | 주로 쓰는 프로토콜 | 느낌                                                   |
| -------------------- | ------------------ | ------------------------------------------------------ |
| Samba                | SMB/CIFS           | 윈도우 **네트워크 드라이브**                           |
| NFS-Kernel-Server    | NFS                | 리눅스끼리 **빠른 공유** (보안·망 설계는 신경 써야 함) |
| vsftpd / ProFTPD     | FTP / SFTP         | 오래됐지만 **대용량 파일** 전송에 아직도 씀            |
| Nextcloud / Owncloud | WebDAV·자체 UI     | **브라우저에서 드라이브** 같은 UI를 빨리 쓰고 싶을 때  |

## Ceph

- **클라우드 S3 비슷한 걸** 회사 안에 직접 깔고 싶을 때 나오는 이름. **최소 노드**가 보통 여러 대.
- 데이터는 **잘라 여러 서버에** 두고, 복제·패리티·장애 시 재구성 같은 게 **제품 철학**에 들어 있음.
- **MON / OSD / MGR / RGW(S3)** … 구성 요소가 많고, **디스크·망·운영** 난이도가 높다고들 함.

---

## ZFS + 파일 서버

- 서버 한 대에 디스크 여러 장 → **ZFS 풀** → 그 위에 dataset → **NFS/SMB** 로 내보냄.
- ZFS가 **RAIDz, 체크섬, 스냅샷, 스크럽** 같은 걸 한꺼번에 담당하는 그림.
- **한 대**가 전부라 서버가 죽으면 서비스도 같이 위험해지고, 꽉 차면 **확장 전략**을 다시 짜야 함. S3 API와는 별개 층.

---

## GlusterFS

- 여러 서버의 `/data` 같은 걸 묶어 **큰 파일시스템 하나**처럼 씀. **파일 단위**로 나눠 담는 느낌에 가깝다 (오브젝트 스토어의 “청크”랑은 다름).
- **Ceph보다 단순하다**는 말을 종종 듣지만, **split-brain** 같은 일은 **자동으로 다 해결**된다고 보긴 어렵다.

## 느낌·구조 비교 (가볍게)

- **MinIO**: OS가 폴더 트리로 수천만 개를 다 관리한다기보다, **객체 key**로 접근하는 쪽에 가깝다. 분산이면 **이레이저 코딩**으로 쪼개 담기도 함.
- **NAS + MinIO**: 바깥에서는 객체 API, 안에는 NAS에 **일반 파일**로 조각이 생길 수 있음. NAS가 이미 **RAID**로 보호 중이면 MinIO **EC를 또** 얹는 건 **중복**일 수 있어서, 부하·구성이 겹친다는 이야기가 나옴.
- **리눅스 + MinIO**: 하드웨어·스펙을 **자유**롭게 잡는 구성이기 쉽다.

| 구분      | NAS (SMB/NFS 중심)         | MinIO (오브젝트)                           |
| --------- | -------------------------- | ------------------------------------------ |
| 쓰는 느낌 | 내 PC **D 드라이브** 비슷  | **웹/S3 API** 비슷                         |
| 트리      | **폴더 계층**이 기본       | **버킷 + 키**(평면에 가깝고 prefix로 정리) |
| 확장      | 슬롯·장비 한계가 자주 이슈 | **노드·디스크** 늘리며 설계                |
| 개발      | 경로·**마운트**            | **S3 SDK** 문서가 많음                     |

| 비교   | 완제품 NAS (예: Synology) | DIY 서버 + MinIO         |
| ------ | ------------------------- | ------------------------ |
| 디스크 | 둘 다 직접 고름           | 둘 다 직접 고름          |
| 본체   | 전용 NAS 케이스           | 일반 서버·케이스         |
| 소프트 | NAS OS 일체형             | 리눅스 + MinIO(오픈소스) |
| 운영   | GUI가 익숙한 편           | 리눅스에 익숙하면 유리   |

## NAS 제품 이름만 정리

**하드웨어 위주**

- **Synology**: 사진·백업·보안 UI 쪽 평이 좋은 편. **확장 유닛**·슬롯 이슈. **고가용**은 두 대 클러스터 같은 **별도** 이야기가 됨.
- **QNAP**: 같은 대략의 라인에서 **CPU/RAM/랜 속도**(예: 2.5GbE)가 나은 경우가 있다. UI는 호불호, 과거 **보안 이슈** 얘기가 있어서 백업 **격리** 얘기가 나옴.

**소프트 위주**

- **TrueNAS**: **ZFS**, 무결성 쪽 신뢰 얘기 많음. **RAM 요구**·설정 난이도는 있다고 함.

## GCP Cloud Storage (한국 리전 — 클래스만)

| 클래스       | 보관(저장) 단가                     | 읽기·조회                                           | 최소 저장 기간 | 한줄                         |
| ------------ | ----------------------------------- | --------------------------------------------------- | -------------- | ---------------------------- |
| **Standard** | 클래스 중 **높은 편**               | 자주 읽는 용도에 맞게 설계된 형태                   | 짧음           | **자주 접근하는** 데이터(핫) |
| **Coldline** | **낮은 편**                         | **읽기(및 복구)에 과금**이 붙음                     | **약 1개월**   | 가끔 꺼내는 보관·백업류      |
| **Archive**  | **가장 낮은 편**에 가깝다고 보면 됨 | **읽기·복구 비용**이 상대적으로 크게 느껴질 수 있음 | **약 1년**     | 장기 아카이브·드물게 읽음    |

- **Coldline / Archive** 는 “GB당 보관료만” 보지 말고 **읽은 양·네트워크·복구(리트리벌)** 까지 같이 본다.
- **최소 저장 기간**을 지키지 않고 삭제하면, 정책에 따라 **남은 기간만큼 저장한 것처럼 추가 비용**이 붙을 수 있다.
