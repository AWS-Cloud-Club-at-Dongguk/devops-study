# 15장 구글 드라이브 설계

구글 드라이브, 드롭박스, 마이크로소프트 원드라이브, 애플 아이클라우드 같은 클라우드 저장 서비스는 널리 쓰인다. 이 장에서는 **구글 드라이브**에 가까운 시스템을 설계한다.

**구글 드라이브**는 파일 저장·동기화 서비스다. 문서·사진·동영상 등을 클라우드에 올리고, PC·스마트폰·태블릿 등 어떤 단말에서든 접근할 수 있어야 하며, 가족·동료 등과 쉽게 공유할 수 있어야 한다.

---

# 1. 문제 이해 및 설계 범위 확정

구글 드라이브 전체를 한 번에 설계하기는 범위가 너무 크므로, 면접에서는 질문으로 범위를 좁힌다.

## 1.1 면접 Q&A로 확정한 내용

| 주제             | 내용                                    |
| ---------------- | --------------------------------------- |
| 가장 중요한 기능 | 파일 업로드·다운로드, 파일 동기화, 알림 |
| 플랫폼           | **웹·모바일 모두** 지원                 |
| 파일당 크기 상한 | **10GB**                                |
| 규모             | DAU **1천만(10 million) 명**            |

## 1.2 기능 요구사항

- 파일 추가(예: 드래그 앤 드롭)
- 파일 다운로드
- 여러 단말 간 **파일 동기화**(한 단말에서 추가하면 다른 단말에도 반영)
- **수정 이력(리비전)** 조회
- 파일 **공유**
- 파일이 **편집·삭제·새로 공유**되었을 때 **알림**

## 1.3 이번 설계에서 다루지 않는 것

- 구글 독스처럼 **여러 사용자가 동시에 편집하는 실시간 협업**은 논의 대상에서 제외한다.

## 1.4 비기능 요구사항

| 항목                        | 설명                                                         |
| --------------------------- | ------------------------------------------------------------ |
| 신뢰성(Reliability)         | 저장 시스템에 매우 중요. **데이터 유실은 용납할 수 없다.**   |
| 빠른 동기화                 | 동기화가 느리면 사용자가 이탈한다.                           |
| 네트워크 대역폭             | 특히 모바일 요금제를 고려해 **불필요한 대역 소비를 줄인다.** |
| 확장성(Scalability)         | 매우 높은 트래픽을 감당할 수 있어야 한다.                    |
| 고가용성(High Availability) | 일부 서버·네트워크 장애가 있어도 **계속 쓸 수 있어야** 한다. |

## 1.5 개략적 규모 추정

| 항목                    | 가정                                                 |
| ----------------------- | ---------------------------------------------------- |
| 가입 사용자             | 약 **5천만 명**                                      |
| DAU                     | **1천만 명**                                         |
| 사용자당 무료 저장 공간 | **10GB**                                             |
| 업로드                  | 사용자당 하루 평균 **2개** 파일, 평균 크기 **500KB** |
| 읽기:쓰기 비율          | **1:1**                                              |

**저장 용량(할당 기준)**  
5천만 × 10GB ≈ **500PB(Petabytes)**

**업로드 API QPS**  
10,000,000 × 2 ÷ (24 × 3600) ≈ **240**  
**피크 QPS** ≈ 평균의 2배 → 약 **480**

# 2. 개략적 설계안 제시 및 동의 구하기

천만 DAU를 감당하는 쪽으로 설계를 발전시킨다.

## 2.1 초기 단일 서버 구성

- **웹 서버**: 파일 업로드·다운로드 처리
- **데이터베이스**: 메타데이터(사용자, 로그인, 파일 정보 등)
- **파일 저장소**: 실제 파일(예: **1TB** 디스크로 시작)

## 2.2 디렉터리 구조(네임스페이스)

파일은 `drive/` 아래에 두고, 그 안에 사용자별 **네임스페이스**(하위 디렉터리)를 둔다. 파일은 **네임스페이스 경로 + 원래 파일 이름** 조합으로 식별한다.

예:

- `user1` → `recipes/chicken_soup.txt`
- `user2` → `football.mov`, `sports.txt`
- `user3` → `best_pic_ever.png`

## 2.3 API 설계

모든 API는 **사용자 인증**이 필요하고 **HTTPS**를 쓴다.

### 파일 업로드

- 작은 파일: **단순 업로드**
- 큰 파일·불안정한 네트워크: **재개 가능 업로드(resumable)**

예시 엔드포인트:

```http
https://api.example.com/files/upload?uploadType=resumable
```

- `uploadType=resumable`
- `data`: 로컬 파일 데이터

**재개 가능 업로드 절차(요약)**

1. 최초 요청으로 **고유 업로드 URL**을 받는다.
2. 데이터를 올리며 진행 상태를 확인한다.
3. 오류 시 **끊긴 지점부터** 이어 올린다.

### 파일 다운로드

```http
https://api.example.com/files/download
```

요청 본문 예:

```json
{
  "path": "/recipes/soup/best_soup.txt"
}
```

### 파일 수정 이력

```http
https://api.example.com/files/list_revisions
```

```json
{
  "path": "/recipes/soup/best_soup.txt",
  "limit": 20
}
```

## 2.4 한 대 서버의 한계와 샤딩

업로드가 늘면 디스크가 가득 찬다. 데이터를 여러 서버로 나누는 **샤딩**을 예로 들 수 있다(예: `user_id % 4`로 서버 1~4에 분산).

다만 샤딩은 **용량 문제**는 완화해도, **한 샤드 서버가 죽으면 그 데이터가 사라지는 위험**은 그대로다.

## 2.5 객체 저장소(예: Amazon S3)로 전환

**S3**는 확장성·가용성·보안·성능을 갖춘 객체 저장 서비스로 소개된다.

- **리전(Region)**: 데이터 센터가 모인 지리적 구역
- **버킷(Bucket)**: 폴더에 가까운 저장 단위
- **복제(다중화)**
  - **동일 리전 내** 복제
  - **여러 리전**에 걸친 복제

## 2.6 로드밸런서와 웹 서버 확장

- **로드밸런서**로 트래픽을 분산하고, 장애 웹 서버는 우회한다.
- 웹 서버는 **수평 확장**(대수 추가)하기 쉬워진다.

## 2.7 메타데이터 DB와 파일 저장소 분리

- **메타데이터 DB**를 파일 저장 서버와 **분리**해 단일 장애점(SPOF)을 줄인다. 가용성·확장을 위해 **복제·샤딩** 정책을 적용한다.
- **파일 저장**은 S3 등을 쓰고, **둘 이상 리전**에 복제해 고가용·내구성을 높인다.

정리하면, 웹/API 서버·메타데이터 DB·파일 저장을 **독립적으로** 두는 구성으로 개선한다.

## 2.8 동기화 충돌

여러 사용자가 **동시에** 같은 파일·폴더를 바꾸면 **동기화 충돌**이 난다.

**처리 전략**: 시스템이 **먼저 처리한 변경**을 성공으로 보고, 그 뒤에 들어온 동시 변경은 **충돌**로 표시한다. 충돌 시 **서버의 최신본**과 **사용자 단말의 로컬 사본**이 공존하므로, 사용자가 **병합**할지 **한쪽으로 덮을지** 결정하게 한다.

충돌 파일 이름 예:

- `SystemDesignInterview`
- `SystemDesignInterview_user_2_conflicted_copy_2019-05-01`

동시 편집·동기화 알고리즘은 더 깊은 주제이므로, 책에서는 참고문헌을 안내한다.

## 2.9 고수준 설계(블록 저장·알림·오프라인 큐 포함)

주요 구성 요소는 다음과 같다.

| 구성 요소                      | 역할                                                                                        |
| ------------------------------ | ------------------------------------------------------------------------------------------- |
| 사용자 단말                    | 웹 브라우저·모바일 앱 등 클라이언트                                                         |
| 로드밸런서                     | API 서버로 요청을 고르게 분산                                                               |
| API 서버                       | 파일 업로드 **외** 대부분(인증, 프로필, 파일 메타데이터 갱신 등)                            |
| 블록 저장소 서버               | 파일을 **블록**으로 쪼개 클라우드 저장소에 업로드                                           |
| 클라우드 저장소                | 블록이 실제로 저장되는 곳(S3 등)                                                            |
| 아카이빙 저장소(콜드 스토리지) | 오랫동안 쓰이지 않은 비활성 데이터 보관                                                     |
| 메타데이터 DB                  | 사용자·파일·블록·버전 등 메타데이터(실파일은 클라우드에만)                                  |
| 메타데이터 캐시                | 자주 쓰는 메타데이터 캐싱                                                                   |
| 알림 서비스                    | pub/sub에 가깝게, 파일 추가·편집·삭제 등 이벤트 시 클라이언트가 최신 상태를 확인하도록 알림 |
| 오프라인 사용자 백업 큐        | 단말이 오프라인이라 즉시 동기화할 수 없을 때, 나중에 맞추기 위한 정보를 쌓아 둠             |

**블록 저장**

- 파일을 여러 **블록**으로 나누고, 블록마다 **고유 해시**를 메타데이터 DB에 둔다.
- 블록은 각각 독립 객체로 S3 같은 곳에 저장한다.
- 파일은 블록을 **원래 순서**로 이어 복원한다.
- 이 설계에서는 블록 최대 크기를 **4MB**로 둔다.

# 3. 상세 설계

## 3.1 일관성과 메타데이터 DB 선택

- DB의 **마스터**가 바뀌면 **캐시된 사본은 무효화**해, 캐시와 DB의 일관성을 맞춘다.
- **관계형 DB**를 택하면 **ACID**로 **강한 일관성**을 비교적 자연스럽게 얻는다. NoSQL은 같은 보장을 **애플리케이션에서 직접** 맞추는 경우가 많다.

## 3.2 메타데이터 DB 스키마(요약)

| 테이블         | 용도·주요 컬럼                                                                                                                                          |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `user`         | 사용자: `user_id`(PK), `user_name`, `created_at` 등. 이름·이메일·프로필 사진 등 기본 정보                                                               |
| `device`       | 단말: `device_id`(PK), `user_id`(FK), `last_logged_in_at`, 모바일 **푸시용 `push_id`** 등. 한 사용자가 여러 단말 가능                                   |
| `namespace`    | 사용자 루트 디렉터리: `id`(PK), `owner_id`(FK), `is_shared`, `created_at`                                                                               |
| `file`         | 파일 최신 정보: `id`(PK), `file_name`, `relative_path`, `is_directory`, `latest_version`, `checksum`, `namespace_id`(FK), `created_at`, `last_modified` |
| `file_version` | 파일 변경 **이력(읽기 위주, 변조 방지)**: `id`(PK), `file_id`(FK), `device_id`(FK), `version_number`, `last_modified`                                   |
| `block`        | 블록: `block_id`(PK), `file_version_id`(FK), `block_order`. 버전은 블록을 순서대로 이어 재구성                                                          |

## 3.3 파일 업로드 흐름

클라이언트는 **동시에 두 갈래** 요청을 보낸다.

1. **메타데이터 추가**(API 서버 → 메타데이터 DB)
2. **파일 데이터 업로드**(블록 저장소 서버 → 클라우드 저장소)

**메타데이터 추가(시작)**

1. 클라이언트가 새 파일 메타데이터를 API 서버에 보낸다.
2. DB에 상태를 **pending(대기)** 으로 둔다.
3. **알림 서비스**에 “새 파일 추가 중”을 알린다.

**이어지는 흐름(요약)**

- 다른 클라이언트는 알림으로 업로드 진행을 알 수 있다.
- 클라이언트는 **블록 저장소 서버**에 파일을 올리고, 서버는 **분할·압축·암호화** 후 클라우드 저장소에 넣는다.
- 클라우드 저장 완료 시 **콜백**으로 API 서버에 알리고, 메타데이터 상태를 **uploaded** 등으로 갱신한다.
- 다시 알림으로 관련 클라이언트에 완료를 알려, **다운로드·동기화**를 이어가게 한다.

**기존 파일 수정** 흐름도 업로드와 **유사**하다.

## 3.4 파일 다운로드·동기화

파일이 추가·편집되면 클라이언트는 변경을 감지해야 한다.

- **온라인**: 알림 서비스가 변경을 **푸시**하고, 새 버전을 가져오라고 한다.
- **오프라인**: 변경 정보를 쌓아 두었다가, 재접속 후 **메타데이터를 당겨오고(pull)** 새 버전을 받는다.

변경을 알게 되면:

1. API 서버에서 **새 메타데이터·변경 이력**을 받고
2. 필요한 **블록**을 블록 저장소 서버를 통해 클라우드에서 받아
3. 단말에서 파일을 **재구성**한다.

**알림 이후 블록까지(단계 정리)**

1. 알림으로 “누군가 파일을 바꿨다”고 전달
2. 클라이언트가 새 메타데이터 요청
3. API 서버가 메타데이터 DB 조회
4. 메타데이터가 클라이언트까지 전달
5. 클라이언트가 블록 다운로드 요청
6. 블록 저장소 서버가 클라우드에서 블록 조회
7. 클라우드가 블록 반환
8. 클라이언트가 블록으로 파일 복원

## 3.5 알림 서비스

파일 일관성을 위해, 로컬에서 수정이 있으면 **곧바로** 다른 클라이언트에 알려 **충돌 가능성을 줄인다.**

| 방식                  | 특징                    |
| --------------------- | ----------------------- |
| 롱 폴링(Long Polling) | 드롭박스 등에서 사용 예 |
| WebSocket             | 지속적 **양방향** 채널  |

이 설계에서는 **롱 폴링**을 택한다.

- 채팅과 달리 **양방향이 필수는 아니다.** 서버→클라이언트 알림이 중심이고 역방향은 상대적으로 드물다.
- WebSocket은 실시간 양방향(채팅 등)에 잘 맞지만, 드라이브의 파일 변경 알림은 **그만큼 잦지 않고** 메시지도 **작다.**

**롱 폴링 동작**

- 클라이언트는 **롱 폴링 연결**을 유지한다.
- 변경이 감지되면 연결이 끊기고, 클라이언트는 메타데이터 서버에서 **최신 상태**를 받는다.
- 다운로드가 끝나거나 **타임아웃**이 나면, 곧바로 다시 롱 폴링 요청을 보내 연결을 재수립한다.

## 3.6 저장소 공간 절약

이력·신뢰성 때문에 여러 버전·데이터센터에 데이터가 쌓여 용량 부담이 크다. 완화책:

1. **중복 제거(De-duplication)**  
   계정 단위로 블록 **해시**를 비교해 동일 블록은 한 번만 저장한다.

2. **지능적 백업 전략**
   - 보관 **버전 개수 상한**을 두고, 넘으면 **가장 오래된 버전**부터 버린다.
   - 자주 저장되는 파일은 쓸데없는 중간 버전이 많으므로, **의미 있는 버전만** 골라 남긴다.

3. **아카이빙(콜드 스토리지)**  
   수개월~수년 접근이 없는 데이터는 **Amazon S3 Glacier** 같은 저비용 저장으로 옮긴다(표준 S3보다 훨씬 싸다고 설명됨).

## 3.7 장애 처리

| 구성 요소               | 대응                                                                                                                                                                                             |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 로드밸런서              | **액티브–패시브** 이중화. **하트비트**로 상태 감시, 주 장애 시 보조가 승격                                                                                                                       |
| 블록 저장소 서버        | 장애 시 **다른 서버가 미완료·대기 작업**을 이어받는다                                                                                                                                            |
| 클라우드 저장소         | 버킷을 **여러 리전에 복제**해, 한 리전 장애 시 다른 리전에서 읽는다                                                                                                                              |
| API 서버                | **무상태**이므로 로드밸런서가 장애 노드로 트래픽을 보내지 않으면 된다                                                                                                                            |
| 메타데이터 캐시         | **복제**해 두고, 노드 하나가 죽으면 다른 곳에서 읽고 실패 노드는 교체                                                                                                                            |
| 메타데이터 DB           | **Primary 장애**: 세컨더리 하나를 승격하고 새 세컨더리를 추가. **세컨더리 장애**: 나머지 세컨더리가 읽기 처리, 실패 노드 교체                                                                    |
| 알림 서비스             | 온라인 사용자마다 롱 폴링 연결 유지. 서버 한 대가 **100만 개 이상 연결**(2012년 드롭박스 발표 인용)을 처리할 수 있다고 하되, 장애 후 **모든 연결을 한꺼번에 다시 맺는 것**은 어렵고 느릴 수 있다 |
| 오프라인 사용자 백업 큐 | **복제**한다. 큐 장애 시 구독 클라이언트는 **백업 큐 구독을 다시** 맺어야 한다                                                                                                                   |

# 4. 마무리

## 4.1 설계가 지향하는 바

**높은 일관성**, **낮은 네트워크 지연**, **빠른 동기화**를 목표로 한다.

핵심은 다음 세 가지다.

1. 파일 **메타데이터 관리**
2. **파일 동기화** 처리
3. **롱 폴링** 기반 **알림 서비스**로 클라이언트 상태를 최신에 가깝게 유지

## 4.2 대안: 클라이언트가 곧바로 클라우드에 업로드

블록 저장소 서버를 거치지 않고 **클라이언트가 직접** 객체 저장소에 올리는 방식도 생각할 수 있다.

| 장점                         | 단점                                                                                                                                                |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| 업로드가 **더 빠를 수 있다** | **분할·압축·암호화**를 **iOS·안드로이드·웹** 등 **모든 클라이언트**에서 구현해야 해 개발이 **복잡**해진다. 블록 서버에 두면 클라이언트는 단순해진다 |

## 4.3 보안·아키텍처 보완

- **암호화 로직을 전부 클라이언트에만 두는 것**은, 클라이언트가 **해킹될 수 있다**는 점에서 적절하지 않을 수 있다.
- **연결 상태 관리** 로직을 알림 서비스와 **분리된 전용 서비스**로 두면, 다른 서비스에서도 **재사용**하기 쉽다.
